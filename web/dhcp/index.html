¿Qué es DHCP?

Asignación automática de configuración de red

DHCP (Dynamic Host Configuration Protocol) permite que un host obtenga automáticamente su configuración de red sin intervención manual: dirección IP, máscara, gateway por defecto, servidores DNS y tiempo de lease.

DHCP asigna:
  • 📍 IP address — única dentro del segmento
  • 🔢 Subnet mask — delimita la red local
  • 🚪 Default gateway — router de salida (punto crítico para MitM)
  • 🔍 DNS servers — resolución de nombres (punto crítico para DNS hijacking)
  • ⏳ Lease time — cuánto tiempo es válida la asignación
  • ➕ Opciones adicionales — WINS, NTP, dominio de búsqueda

El proceso DORA

Discover → Offer → Request → Acknowledge

Cliente (0.0.0.0)               Servidor DHCP (192.168.1.1)
      │                                    │
      │── DISCOVER (broadcast) ───────────►│  src=0.0.0.0:68 dst=255.255.255.255:67
      │   "¿Hay algún DHCP en la red?"     │  MAC: aa:bb:cc:dd:ee:ff
      │                                    │
      │◄──── OFFER ─────────────────────── │  "Te ofrezco 192.168.1.105/24"
      │   IP=192.168.1.105                 │  GW=192.168.1.1  DNS=8.8.8.8
      │   Lease=86400s (24h)               │
      │                                    │
      │── REQUEST (broadcast) ────────────►│  "Acepto la oferta de 192.168.1.1"
      │   Requested IP=192.168.1.105       │  (broadcast para que otros DHCP vean)
      │                                    │
      │◄──── ACK ───────────────────────── │  "Confirmado. IP tuya por 86400s"
      │   Configuración completa           │
      │                                    │
  CONFIGURA interfaz con IP+GW+DNS asignados
Sin autenticación: DHCP no tiene mecanismo de autenticación nativo. Cualquier máquina en la red puede responder OFFERS. El cliente acepta la primera OFFER que llegue — o la más rápida.

Ataque 1: DHCP Starvation

Agotamiento del pool de IPs — DoS

El atacante envía miles de DHCP DISCOVER con MACs falsas aleatorias. Cada DISCOVER reserva una IP del pool. Cuando el pool se agota, clientes legítimos no pueden obtener IP y quedan sin red.

Atacante (herramienta: Yersinia / dhcpstarv)
  │
  ├── DISCOVER  src_mac=aa:bb:cc:11:11:11  "dame IP"
  ├── DISCOVER  src_mac=aa:bb:cc:22:22:22  "dame IP"
  ├── DISCOVER  src_mac=aa:bb:cc:33:33:33  "dame IP"
  │   ... × 254 veces en segundos ...
  │
  ▼  POOL AGOTADO

  Cliente legítimo:
  ├── DISCOVER  src_mac=aa:bb:cc:real:mac  "dame IP"
  └── ← (sin respuesta — pool vacío)  → ¡Sin red!

MITRE ATT&CK: T1498 — Network Denial of Service · CAPEC-61

Ataque 2: Rogue DHCP Server

Man-in-the-Middle mediante configuración de red maliciosa

El atacante levanta un servidor DHCP en la red (a veces después de un starvation). Los clientes reciben configuración de red con el gateway y DNS del atacante. Todo el tráfico pasa por el atacante antes de salir a Internet.

NORMAL:   Cliente → [GW real 192.168.1.1] → Internet

ROGUE:    Cliente → [GW atacante 192.168.1.200] → [GW real] → Internet
                          ↑
                    ARP, sniff, inyección, SSL strip
                    DNS falso → sinkhole / phishing

DHCP OFFER malicioso:
  IP=192.168.1.105  ← IP válida (para no levantar sospechas)
  Gateway=192.168.1.200  ← IP del atacante
  DNS=10.0.0.1  ← DNS malicioso del atacante
  Lease=300s  ← lease corto para renovar rápido
Impacto: captura de credenciales, tokens de sesión, datos sensibles. Redirección silenciosa a sitios de phishing. El usuario no nota nada — la navegación parece funcionar.

MITRE ATT&CK: T1557.003 — Adversary-in-the-Middle: DHCP Spoofing

Ataque 3: DHCP Starvation + Rogue combinado

El ataque completo en dos fases

FASE 1 — STARVATION (silenciar el DHCP legítimo):
  └─ Enviar DISCOVER con cientos de MACs falsas
     → Pool del DHCP legítimo = agotado
     → Clientes que piden IP no obtienen respuesta del DHCP real

FASE 2 — ROGUE DHCP (tomar el control):
  └─ Levantar DHCP falso que SOLUCIONA el problema
     → Responde rápido con configuración "válida"
     → Desesperados por red, los clientes aceptan la oferta
     → Todos los nuevos hosts → gateway=atacante

RESULTADO: MitM total sin alertas evidentes
  → Clientes con red funcional (no sospechan)
  → Todo el tráfico pasa por el atacante
  → Atacante puede HTTPS strip, SSLstrip, capturar credenciales

DETECCIÓN:
  → Dos servidores DHCP en el mismo segmento (IOC fuerte)
  → DHCP leases con gateway distinto al esperado
  → Súbita tasa alta de DISCOVER desde una MAC

Defensa: DHCP Snooping

El switch como árbitro de DHCP

DHCP Snooping es una función de seguridad de switches que clasifica los puertos en trusted (solo hacia servidores DHCP legítimos) y untrusted (hacia clientes).

Switch con DHCP Snooping habilitado:

[Puerto Uplink — TRUSTED]         [Puertos de acceso — UNTRUSTED]
  ↕ DHCP OFFER permitido              ↕ Solo DISCOVER/REQUEST (cliente)
  ↕ DHCP ACK permitido                ✗ OFFER/ACK bloqueado (evita rogue)
  ↕ DHCP DISCOVER permitido           ✗ MACs falsas → rate limit + drop

Cisco IOS:
  ip dhcp snooping
  ip dhcp snooping vlan 10
  interface GigabitEthernet0/1   ← hacia clientes
    ip dhcp snooping limit rate 15   ← máx 15 paquetes DHCP/s
  interface GigabitEthernet0/24  ← hacia servidor DHCP real
    ip dhcp snooping trust

DHCP Snooping Binding Table:
  MAC address   IP addr        Lease   VLAN  Interface
  aa:bb:cc...   192.168.1.105  86400s  10    Gi0/5
  → Usada por ARP Inspection + IP Source Guard para validar tráfico

Detección y mitigación completa

Checklist para SOC + networking

  • 🔒 DHCP Snooping en todos los switches de acceso — bloquea OFFER desde puertos untrusted
  • 🔍 Dynamic ARP Inspection — valida ARP contra la DHCP Snooping Binding Table
  • 🛡️ IP Source Guard — filtra paquetes cuya IP/MAC no coincide con la tabla de snooping
  • 📊 Monitorear Zeek dhcp.log — alertar ante múltiples servidores DHCP o tasa alta de DISCOVER desde misma MAC base
  • ⚠️ Rate limiting DHCP — máx 10-15 paquetes DHCP/s por puerto
  • 🔐 802.1X / NAC — autenticación antes de acceso a red (elimina el problema de raíz)
  • 📋 Auditar leases — comparar gateway en leases vs gateway esperado; alertar ante discrepancias
EJ-01 Identificar fase DORA Fácil

Para cada paquete capturado, indicá qué fase DORA es y qué información contiene:

Paquete 1: src=0.0.0.0:68  dst=255.255.255.255:67
  msg_type=1  chaddr=de:ad:be:ef:00:01
  requested_ip=0.0.0.0  server_id=0.0.0.0

Paquete 2: src=192.168.1.1:67  dst=255.255.255.255:68
  msg_type=2  yiaddr=192.168.1.110
  options: lease=86400, gw=192.168.1.1, dns=8.8.8.8

Paquete 3: src=0.0.0.0:68  dst=255.255.255.255:67
  msg_type=3  chaddr=de:ad:be:ef:00:01
  requested_ip=192.168.1.110  server_id=192.168.1.1

Paquete 4: src=192.168.1.1:67  dst=192.168.1.110:68
  msg_type=5  yiaddr=192.168.1.110  lease=86400
Paquete 1: DISCOVER → Fase D: el cliente (MAC de:ad:be:ef:00:01) busca servidores DHCP → src=0.0.0.0 porque aún no tiene IP → dst=255.255.255.255 (broadcast) para alcanzar cualquier DHCP → msg_type=1 = DHCPDISCOVER Paquete 2: OFFER → Fase O: el servidor (192.168.1.1) ofrece la IP 192.168.1.110 → yiaddr = "your IP address" = IP ofrecida al cliente → Incluye parámetros: lease 24h, gateway, DNS → msg_type=2 = DHCPOFFER Paquete 3: REQUEST → Fase R: el cliente acepta la oferta y la solicita formalmente → server_id=192.168.1.1: dice explícitamente a qué servidor acepta (otros servidores DHCP que también ofrecieron pueden retirar su oferta) → Sigue siendo broadcast para que todos los DHCP se enteren → msg_type=3 = DHCPREQUEST Paquete 4: ACK (Acknowledge) → Fase A: el servidor confirma la asignación definitiva → yiaddr=192.168.1.110: la IP es tuya → lease=86400s: válida por 24h → msg_type=5 = DHCPACK → A partir de aquí el cliente configura su interfaz DATO: msg_type=6 sería DHCPNACK (servidor rechaza la solicitud, por ejemplo si la IP ya fue asignada a otro)
EJ-02 ¿Por qué DHCP Starvation no siempre necesita starvation previo? Fácil

Un atacante quiere hacer Rogue DHCP sin ejecutar el starvation primero. ¿Es posible? ¿Bajo qué condiciones funciona y cuándo no? ¿Qué ventaja tiene hacerlo con starvation previo?

SÍ, el Rogue DHCP funciona SIN starvation previo cuando: → El atacante responde más rápido que el servidor legítimo → El cliente acepta la PRIMERA oferta que recibe → Si el rogue DHCP está en la misma LAN, tiene menor latencia que un servidor en otro segmento o cargado con muchas solicitudes CONDICIONES DONDE FUNCIONA SIN STARVATION: 1. Red con muchos clientes: el servidor DHCP legítimo está ocupado y tarda más en responder → el rogue gana la carrera 2. Rogue en el mismo switch: menor latencia física 3. Lease corto (300s): las renovaciones son frecuentes → más oportunidades 4. Nuevos dispositivos conectándose: primera vez en la red CONDICIONES DONDE NO FUNCIONA: → El cliente ya tiene un lease válido y no busca nuevo DHCP → El servidor legítimo es más rápido y el cliente recibe su oferta primero → El switch tiene DHCP Snooping activo (bloquea el OFFER del rogue) VENTAJA DE HACERLO CON STARVATION PREVIO: → El servidor legítimo NO PUEDE responder (pool vacío, sin IPs para ofrecer) → El rogue es el único que puede dar IP → el cliente no tiene otra opción → Garantiza el 100% de las víctimas en vez de depender de velocidad → Mucho más efectivo en redes bien configuradas CASO ESPECIAL — renovación de lease: Al 50% del lease (T1), el cliente envía DHCPREQUEST al servidor original. Si el server legítimo no puede renovar (pool agotado), el rogue puede interceptar con DHCPNACK + nueva OFFER maliciosa.

EJ-05 ¿Cómo saber si ya hay un MitM activo vía DHCP rogue? Difícil

Ya pasó el ataque y sospechás que hay un MitM activo. ¿Cómo confirmás que las víctimas están siendo interceptadas? ¿Cómo determinás el alcance (cuántos hosts afectados)?

Aislar 192.168.1.99 del switch inmediatamente 2. Lista de hosts afectados → forzar renovación DHCP (ipconfig /release + /renew) 3. Investigar qué tráfico capturó el atacante (análisis forense de .99) 4. Notificar usuarios: posible exposición de credenciales HTTP/no-HTTPS 5. Forzar cambio de contraseñas de sistemas accedidos desde hosts afectados 6. Revisar si el atacante tenía acceso físico o era un host comprometido
EJ-06 DHCP Option Spoofing — identificar la opción maliciosa Medio

En un DHCP ACK capturado con Wireshark ves estas opciones: Option 1 (Subnet Mask): 255.255.255.0 | Option 3 (Router): 10.0.0.99 | Option 6 (DNS): 10.0.0.99 | Option 15 (Domain): evil-corp.local | Option 51 (Lease): 300s. El gateway legítimo de la red es 10.0.0.1. (a) ¿Qué opciones son maliciosas? (b) ¿Qué ataque representa cada una? (c) ¿Por qué el lease time de 300s es sospechoso? (d) ¿Cómo detectarías esto con Zeek?

(a) OPCIONES MALICIOSAS: Option 3 (Router/Gateway): 10.0.0.99 esperado: 10.0.0.1 → MitM activo Option 6 (DNS): 10.0.0.99 DNS del atacante → DNS spoofing Option 15 (Domain): evil-corp.local dominio falso → NTLM hash capture (Option 1 es legítima; Option 51 con 300s es sospechosa pero no definitiva) (b) ATAQUE POR OPCION: Option 3 → Gateway spoofing (T1557.003): todo el tráfico pasa por 10.0.0.99 El atacante intercepta HTTP/HTTPS, credenciales, cookies Option 6 → DNS spoofing: el cliente resuelve nombres via 10.0.0.99 Permite redirigir paypal.com, intranet, etc. a IPs del atacante Option 15 → Domain spoofing: el cliente intenta autenticarse en evil-corp.local Responder captura los hashes NTLM enviados automáticamente (c) LEASE TIME DE 300s (5 MINUTOS): El lease corporativo estándar es 8-24h. Un lease corto fuerza que el cliente renueve DHCP cada 5 minutos, dando al atacante múltiples oportunidades de inyectar opciones. También dificulta rastrear el inicio del ataque en logs. (d) DETECCION CON ZEEK (dhcp.log): # Servidor DHCP no autorizado zeek-cut server_addr | sort -u → Si ves 10.0.0.99 además de 10.0.0.1 = rogue confirmado # Gateway sospechoso zeek-cut mac assigned_ip gateway | awk '$3 != "10.0.0.1"' # DNS externo o interno inusual zeek-cut mac dns | grep -v "10.0.0.1\|8.8.8.8" # Lease time anomalo (menos de 30 min) zeek-cut mac lease_time | awk '$2 < 1800' # Dominio inesperado zeek-cut mac domain | grep -v "empresa.local" MITIGACION: - DHCP Snooping: descarta DHCP ACK desde puertos untrusted - DAI: valida que MAC+IP correspondan al binding de snooping - DNS over HTTPS desde los endpoints para ignorar la Option 6 - GPO Windows: dominio de inicio de sesión fijo, no aceptar dominios via DHCP - MITRE: T1557.003 + T1557.001 (LLMNR si usa Responder)

Herramienta · Análisis

Zeek dhcp.log — consultas

# Encontrar servidores DHCP no autorizados
zeek-cut server_addr < dhcp.log | sort | uniq -c | sort -rn

# Hosts con gateway sospechoso
zeek-cut mac assigned_ip gateway < dhcp.log \
  | awk '$3 != "192.168.1.1"'

# Starvation: muchas MACs del mismo OUI
zeek-cut ts mac < dhcp.log \
  | awk '{print substr($2,1,8)}' | sort | uniq -c | sort -rn

🔒 CVEs Relevantes
CVE

CVE-2018-1111 — dhclient RCE (RHEL/CentOS)

NetworkManager en RHEL 6/7 ejecutaba código arbitrario vía un script del hook de dhclient al recibir opciones DHCP maliciosas. CVSS 7.5. Un rogue DHCP podía comprometer hosts Linux al renovar lease.

→ nvd.nist.gov/CVE-2018-1111
CVE

CVE-2019-0547 — Windows DHCP Server RCE

Vulnerabilidad de ejecución remota de código en el servidor DHCP de Windows Server 2019. Un atacante sin autenticación podía ejecutar código arbitrario enviando paquetes DHCP especialmente diseñados. CVSS 9.8.

→ nvd.nist.gov/CVE-2019-0547
CVE

CVE-2011-0997 — ISC dhclient script injection

El cliente DHCP de ISC (usado en Debian/Ubuntu) no sanitizaba el nombre de host enviado por el servidor DHCP. Un rogue DHCP podía inyectar comandos shell en el script de actualización del hostname.

→ nvd.nist.gov/CVE-2011-0997
