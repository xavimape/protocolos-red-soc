¿Qué es TCP?

El protocolo de transporte confiable de Internet

TCP (Transmission Control Protocol) es el protocolo de capa de transporte que garantiza entrega ordenada, sin errores y sin duplicados de datos entre dos hosts. Opera sobre IP y es la base de HTTP, HTTPS, SSH, SMTP, FTP y la mayoría de los protocolos de aplicación.

Características clave de TCP:
  • 🔗 Orientado a conexión — requiere establecer conexión antes de enviar datos (3-way handshake)
  • ✅ Entrega garantizada — ACK por cada segmento; retransmite si no hay confirmación
  • 📋 Ordenamiento — sequence numbers aseguran que los datos lleguen en orden
  • 🚦 Control de flujo — sliding window evita saturar al receptor
  • 🔧 Control de congestión — slow start, AIMD, CUBIC

El Three-Way Handshake

Cómo se establece una conexión TCP

Cliente (192.168.1.100)          Servidor (93.184.216.34:443)
       │                                    │
       │──── SYN (seq=1000) ───────────────►│  [1] Cliente inicia
       │                                    │      SYN_SENT
       │◄─── SYN-ACK (seq=5000, ack=1001) ─│  [2] Servidor confirma
       │                                    │      SYN_RECEIVED
       │──── ACK (ack=5001) ───────────────►│  [3] Conexión establecida
       │                                    │      ESTABLISHED
       │════════ DATA TRANSFER ═════════════│
       │                                    │
       │──── FIN ──────────────────────────►│  [4] Cierre (4-way)
       │◄─── ACK ───────────────────────────│
       │◄─── FIN ───────────────────────────│
       │──── ACK ──────────────────────────►│  CLOSED
Flags TCP: SYN (sync), ACK (acknowledge), FIN (finish), RST (reset), PSH (push), URG (urgent). La combinación de flags en cada paquete indica el estado de la conexión.

  • Window Size 0 → RST programado o intento de denegación

  • Flags inválidas (SYN+FIN, NULL) → escaneo evasivo (Xmas scan, NULL scan)

Ataque 1: SYN Flood

El DDoS clásico de capa de transporte

El atacante envía una ráfaga de paquetes SYN con IPs de origen falsificadas (spoofed). El servidor reserva recursos para cada "conexión" semi-abierta en la Transmission Control Block (TCB) table y espera el ACK final que nunca llega.

Impacto:
  • La TCB table se satura → el servidor rechaza conexiones legítimas
  • Denegación de servicio efectiva con relativamente poco ancho de banda del atacante
  • Las IPs spoofed dificultan el bloqueo por origen

Mitigación: SYN cookies (el servidor no reserva estado hasta recibir ACK), rate limiting de SYN por IP, firewall stateful, scrubbing centers.

MITRE ATT&CK: T1498.001 — Direct Network Flood

5 / 10

Ataque 2: Port Scanning

Reconocimiento de servicios expuestos

El atacante envía paquetes TCP a múltiples puertos para identificar servicios activos. Los distintos tipos de scan son detectables con patrones de flags:

  • SYN scan (-sS): envía SYN, espera SYN-ACK (abierto) o RST (cerrado). Más sigiloso: no completa el handshake.
  • Connect scan (-sT): completa el handshake. Queda en logs de la aplicación.
  • NULL scan (-sN): envía paquete sin flags. Abierto→ no responde, Cerrado→ RST.
  • Xmas scan (-sX): SYN+FIN+URG+PSH. Mismo comportamiento que NULL.
  • FIN scan (-sF): solo FIN. Evasión de algunos firewalls stateless.
IOC en logs: una IP enviando SYN a más de 15–20 puertos distintos en <1 minuto = port scan casi seguro.

MITRE ATT&CK: T1046 — Network Service Discovery

6 / 10

Ataque 3: TCP Session Hijacking

Robo de una sesión TCP activa

El atacante en posición MITM intercepta una sesión TCP activa y la toma control inyectando paquetes con los sequence numbers correctos. Pasos:

  1. Sniff del tráfico TCP para obtener seq/ack numbers actuales
  2. Inyectar paquetes con src=víctima, seq=esperado → el servidor lo acepta
  3. Desincronizar al cliente legítimo (RST o desynch de sequence)
  4. El atacante tiene control total de la sesión
Señal en SOC: RST inesperados en conexiones activas, retransmisiones anómalas, cambio de sequence numbers sin causa de red.

Detección y mitigación

Resumen operativo para SOC

  • 🛡️ SYN Cookies — el kernel no reserva TCB hasta recibir ACK válido. Activo por defecto en Linux.
  • 🔥 Firewall Stateful — solo permite paquetes que pertenecen a una conexión establecida.
  • ⏱️ Rate Limiting SYN — iptables/nftables: máx N SYN/s por IP de origen.
  • 🕵️ IDS signatures — Suricata detecta NULL/Xmas scans por combinación de flags inválidas.
  • 📊 NetFlow / IPFIX — baseline de conexiones TCP por destino; anomalías → alerta SIEM.
  • 🚫 Blackholing / Scrubbing — durante DDoS volumétrico, desviar tráfico a centros de limpieza.
Regla Sigma — Port Scan Detection:
selection: conn_state: "S0" AND unique_dst_ports > 20 AND timeframe: 60s
groupby: src_ip

Ataque 5: ACK Flood DoS

Saturar la tabla de estados TCP con paquetes ACK espurios

El ACK Flood envía millones de paquetes ACK con IPs spoofed y números de secuencia aleatorios. El servidor debe buscar cada ACK en su tabla de conexiones para determinar si pertenece a una sesión válida. Con suficiente volumen, la CPU se satura procesando paquetes inválidos.

MECANISMO:
  - Paquetes ACK válidos: el servidor los procesa rápido (sesión conocida)
  - Paquetes ACK inválidos: el servidor responde con RST → doble gasto de CPU
  - Sin sesión previa → el firewall stateful debe verificar cada uno
  - Volumétrico: fácil de amplificar con botnet

COMANDO (hping3):
  hping3 -A --rand-source --flood -p 80 10.0.0.1
  # -A    = flag ACK activado
  # --rand-source = IP origen aleatoria (spoofed)
  # --flood = máxima velocidad posible

VS SYN FLOOD:
  SYN Flood: crea half-open connections, agota tabla TCP
  ACK Flood: no crea estado, pero fuerza búsquedas en tabla
  ACK Flood es más difícil de filtrar (ACK es tráfico normal de vuelta)

DETECCIÓN:
  Zeek: paquetes ACK sin SYN previo correspondiente
  iptables: -m state --state ESTABLISHED → bloquear ACK sin estado
  Netflow: ratio ACK/SYN anómalo (>>1 en condiciones normales es ok, >>100 es sospechoso)
  SIEM: alerta en tasa de RST responses desde el servidor

MITIGACIÓN:
  Rate limiting en ACK sin conexión establecida
  SYN cookies + connection tracking en firewall
  Anycast + scrubbing center ante ataques volumétricos

MITRE ATT&CK: T1498.001 — Direct Network Flood

Ataque 6: TCP IP Spoofing

Falsificar la IP origen para hacerse pasar por otro host

IP Spoofing en TCP consiste en forjar la dirección IP origen de los paquetes. En redes modernas es difícil para ataques bidireccionales (no se recibe la respuesta), pero se combina con ARP poisoning para MitM, o se usa en ataques blind para inyectar datos o resetear conexiones.

TIPOS DE TCP SPOOFING:

1. BLIND SPOOFING (sin respuesta visible):
   - El atacante no ve la respuesta del target
   - Debe predecir el ISN (Initial Sequence Number)
   - Requiere silenciar la IP suplantada (iptables DROP)
   - scapy: send(IP(src="victim",dst="target")/TCP(flags="S",seq=ISN_guess))

2. MitM + ARP POISONING (con respuesta visible):
   - ARP Poisoning redirige el tráfico a través del atacante
   - El atacante ve las respuestas → spoofing bidireccional
   - Permite inyectar comandos en sesiones Telnet, HTTP no cifrado
   - arpspoof + scapy/inject para modificar tráfico en-vuelo

3. SESSION RESET:
   - Sniffar SEQ number de una sesión activa
   - Enviar RST con IP spoofed y SEQ dentro de la ventana
   - Termina la conexión sin ser parte de ella (ver RST Injection)

HERRAMIENTAS:
  scapy:    send(IP(src="1.2.3.4",dst="5.6.7.8")/TCP(flags="S"))
  hping3:   hping3 --spoof 1.2.3.4 -S -p 80 5.6.7.8
  tcpkill:  terminar conexiones TCP activas en la red local

DETECCIÓN:
  uRPF (Unicast Reverse Path Forwarding): bloquea IPs con ruta asimétrica
  Ingress filtering (BCP38): ISPs deben filtrar paquetes con IPs propias
  ARP inspection dinámica: detecta ARP poisoning
  Zeek: conn.log — IPs que no corresponden a la MAC registrada

EJ-06 ACK Flood: ¿cómo difiere del SYN Flood? Medio

Explicá las diferencias técnicas entre SYN Flood y ACK Flood. ¿Por qué los SYN cookies no mitigan el ACK Flood? ¿Qué recurso del servidor atacan respectivamente? ¿Cómo detectarías un ACK Flood en Zeek conn.log? ¿Qué regla Sigma escribirías?

DIFERENCIAS TÉCNICAS: SYN Flood: paquetes con flag SYN → llena la SYN backlog (half-open connections) ACK Flood: paquetes con flag ACK → el kernel procesa cada ACK buscando la sesión → CPU POR QUÉ SYN COOKIES NO AYUDAN: SYN cookies resuelven el problema del backlog (SYN Flood) Los ACK packets pasan directo al TCP stack → el kernel gasta CPU buscando la conexión No hay backlog que proteger — el ataque es volumétrico puro RECURSO ATACADO: SYN Flood → memoria (backlog de half-open connections) ACK Flood → CPU (interrupt handling, softirq, TCP state lookup) DETECCIÓN EN ZEEK conn.log: Buscar: proto=tcp, conn_state=S0 (ACKs sin sesión previa) alto volumen de paquetes con RST inmediato del servidor REGLA SIGMA: selection: proto: tcp tcp_flags|contains: 'A' # ACK flag conn_state: 'RSTO' # RST desde origin (sin sesión) condition: selection | count() by dst_ip > 10000 | timeframe: 10s
EJ-07 TCP Spoofing: blind vs. non-blind Difícil

¿Cuál es la diferencia entre TCP Spoofing ciego y no-ciego? ¿Por qué los ISNs (Initial Sequence Numbers) aleatorios dificultan el ataque? ¿Qué condición de red se necesita para el ataque no-ciego? ¿Cómo se usa TCP Spoofing para inyectar RST y terminar sesiones BGP?

BLIND TCP SPOOFING: El atacante NO está en el path (no puede ver las respuestas) Necesita adivinar el ISN (Initial Sequence Number) del servidor Históricamente posible cuando los ISNs eran predecibles (e.g. incrementos fijos) Hoy: RFC 6528 → ISNs aleatorios → ataque prácticamente inviable NON-BLIND TCP SPOOFING: El atacante ESTÁ en el path (MitM, mismo segmento de red) Puede ver las respuestas → conoce los sequence numbers reales Permite: inyectar datos, RST injection, session hijacking ISN RANDOMIZATION (RFC 6528): ISN = F(secreto, src_ip, dst_ip, src_port, dst_port, timestamp) Sin adivinar el ISN correcto, el TCP stack ignora el paquete spoofed RST INJECTION vs BGP: BGP usa TCP puerto 179 con sesiones de larga duración (horas/días) Atacante envía RST con src=routerA, dst=routerB, window matching Si el seq number cae en la ventana → la sesión BGP cae → routing disruption Caso real: CVE-2004-0230 — RST injection contra sesiones BGP Mitigación: TCP MD5 (RFC 2385), GTSM (RFC 5082), filtros BCP38
