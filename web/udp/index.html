¿Qué es UDP?

El protocolo de transporte sin conexión

UDP (User Datagram Protocol) es el protocolo de capa de transporte diseñado para velocidad sobre confiabilidad. No establece conexión, no garantiza entrega, no ordena paquetes y no retransmite. Un datagrama UDP se envía y listo.

¿Cuándo usar UDP?
  • 🎮 Gaming / VoIP / streaming — la latencia importa más que la pérdida de un paquete
  • 🔍 DNS — query/response rápido; TCP solo para zone transfers y respuestas >512 bytes
  • ⏱️ NTP — sincronización de tiempo; un paquete basta
  • 📡 DHCP / SNMP / Syslog / TFTP — protocolos simples que no necesitan garantía
  • 🚀 HTTP/3 (QUIC) — UDP con confiabilidad implementada en la capa de aplicación

La cabecera más simple de la pila TCP/IP

│  Source Port   │   Dest Port    │  (2 + 2 bytes)
├────────────────┼────────────────┤

TCP header: 20–60 bytes
UDP header:       8 bytes   ← 10x más pequeña
Sin estado: UDP no mantiene ninguna tabla de conexiones. Cada datagrama es independiente. Esto lo hace ideal para el spoofing de IP de origen — el servidor no puede verificar si el origen es real.

Ataque 1: UDP Flood

DDoS volumétrico directo

El atacante envía una avalancha de datagramas UDP hacia puertos aleatorios del objetivo. Como no hay ningún estado de conexión, no hay defensa equivalente a SYN cookies. El servidor:

  1. Recibe el datagrama en puerto X
  2. Verifica si hay servicio escuchando en X → no hay
  3. Genera respuesta ICMP "Port Unreachable" → consume CPU y ancho de banda
  4. Repite para los siguientes 10,000 paquetes/segundo
Impacto: agotamiento de ancho de banda de salida (por ICMP unreachable), CPU alta, links saturados. Muy fácil de generar con herramientas como hping3 o botnets.

MITRE ATT&CK: T1498.001 — Direct Network Flood

Ataque 3: UDP Port Scan

Reconocimiento de servicios UDP — más lento pero necesario

El escaneo UDP es más difícil que TCP porque no hay respuesta para puertos abiertos — el servicio puede simplemente ignorar el datagrama. La lógica de detección se invierte:

  • 🟢 Puerto abierto: el servicio responde (DNS responde a query, NTP responde a monlist)
  • ❓ Puerto abierto pero sin respuesta: el servicio ignora el datagrama
  • 🔴 Puerto cerrado: el OS responde ICMP "Port Unreachable"
  • 🟡 Puerto filtrado: no hay respuesta (firewall dropea silenciosamente)
IOC: muchos ICMP "Port Unreachable" desde el mismo destino en un período corto = probable UDP scan en curso. También: datagramas hacia puertos inusuales (7, 19, 69, 111, 137...).

MITRE ATT&CK: T1046 — Network Service Discovery

Detección y mitigación

Resumen operativo para SOC

  • 🚫 Deshabilitar NTP monlist: ntpdc -c "monlist off" o actualizar NTP ≥4.2.7p26
  • 🔒 BCP38 / uRPF: los ISPs deben filtrar paquetes con IP origen spoofed (evita amplificación)
  • 📊 Rate limiting ICMP unreachable: limitar respuestas ICMP para no agotarse respondiendo scans
  • 🌐 Anycast + scrubbing: ante UDP Flood volumétrico, desviar tráfico a centros de limpieza
  • 🔥 ACLs en borde: bloquear UDP entrante a puertos de servicios no expuestos (CharGen/19, QOTD/17)
  • 📈 Baseline NetFlow: alertar cuando el volumen UDP hacia un host supera N Mbps o N pps

Ataque 5: Memcached Amplification DDoS

Factor de amplificación ×51,000x — el mayor registrado

Memcached es un sistema de caché en memoria. Si está expuesto en UDP (puerto 11211), un request de 15 bytes puede generar una respuesta de hasta 750 KB. En 2018, GitHub recibió 1.7 Tbps usando exactamente esta técnica.

MECANISMO:
  Payload enviado al Memcached (15 bytes):
    stats

  Respuesta Memcached (hasta 750,000 bytes):
    STAT pid 12345
 STAT uptime 3600
 ... (miles de líneas)

FACTOR DE AMPLIFICACIÓN: ×51,200x (en casos reales reportados)
ANCHO DE BANDA TÍPICO:
  1,000 servers × 750 KB × 8 bits = 6 Tbps sobre la víctima

COMANDO (shodan + exploit):
  shodan search "port:11211 product:Memcached"  → encontrar targets
  echo -e "stats
" | nc -u TARGET 11211
  # Spoofear IP origen = víctima para redirigir tráfico

DETECCIÓN:
  Netflow: UDP port 11211, respuestas >>requests (ratio anómalo)
  Zeek: conn.log, bytes enviados vs recibidos
  Shodan Monitor: alerta si tu IP aparece como Memcached expuesto

MITIGACIÓN:
  Deshabilitar UDP en Memcached: --listen=127.0.0.1 -U 0
  Firewall: bloquear port 11211 externamente
  Nunca exponer Memcached a Internet (solo localhost o VPN)

Ataque 6: SSDP Amplification DDoS

Dispositivos IoT y UPnP como reflectores de ataques DDoS

SSDP (Simple Service Discovery Protocol) es parte de UPnP. Los dispositivos IoT, routers y smart TVs responden a M-SEARCH con descripciones XML completas. Un request de ~110 bytes genera una respuesta de ~400 bytes (×75x) que se redirige a la víctima.

MECANISMO:
  M-SEARCH request (110 bytes) → enviado a dispositivo UPnP
  Respuesta XML (~400 bytes) → redirigida a víctima (IP spoofed)

  REQUEST:
    M-SEARCH * HTTP/1.1
    HOST: 239.255.255.250:1900
    MAN: "ssdp:discover"
    ST: ssdp:all
    MX: 1

  RESPUESTA (ejemplo):
    HTTP/1.1 200 OK
    CACHE-CONTROL: max-age=1800
    LOCATION: http://192.168.1.1:49152/rootDesc.xml
    USN: uuid:device-UUID::upnp:rootdevice
    SERVER: Linux/2.6 UPnP/1.0 MiniUPnPd/1.9

FACTOR: ×30-75x
VECTORES CONOCIDOS: routers domésticos, smart TVs, cámaras IP, NAS

ESCANEO + EXPLOIT:
  masscan -p1900 --rate 10000 0.0.0.0/0 -oL ssdp_devices.txt
  python3 ssdp_amp.py --victims ssdp_devices.txt --target VICTIM_IP

DETECCIÓN:
  Netflow: UDP src port 1900, respuestas desde múltiples IPs
  Volumen anómalo de tráfico UDP hacia una única IP destino
  IDS: regla para respuestas SSDP fuera de red local

MITIGACIÓN:
  Deshabilitar UPnP en routers (especialmente la interfaz WAN)
  Firewall: bloquear UDP port 1900 hacia/desde Internet
  ISP: BCP38 para evitar spoofing en redes de origen

EJ-01 UDP o TCP — ¿cuál usarías? Fácil

Para cada escenario, justificá si usarías UDP o TCP y por qué:

  1. Transferencia de un archivo de configuración de 5 MB entre servidores
  2. Streaming de video en vivo para 10,000 espectadores
  3. Sincronización de hora entre servidores del datacenter
  4. Login de usuario en una app bancaria
  5. Envío de logs de syslog desde 500 equipos a un SIEM
1. Archivo 5MB → TCP Razón: debe llegar completo y en orden. Un solo paquete faltante corrompe el archivo. TCP retransmite automáticamente. 2. Video en vivo → UDP (con QUIC/RTP encima) Razón: un frame perdido es preferible a la latencia de retransmisión. Si se pierde, se muestra el frame anterior o se interpola. La latencia baja importa más que la perfección. 3. Sincronización NTP → UDP/123 Razón: NTP necesita latencia mínima para calcular el offset de tiempo. Un solo paquete basta. La precisión se degrada con retransmisiones TCP. 4. Login bancario → TCP (HTTPS sobre TCP) Razón: la integridad y el orden son críticos. Las credenciales deben llegar completas. TLS sobre TCP garantiza esto + cifrado. 5. Syslog desde 500 equipos → UDP/514 (o TCP/514 para críticos) Razón: alta velocidad, volumen masivo, pérdida ocasional aceptable. UDP evita saturar el SIEM con retransmisiones. Para logs críticos (seguridad, compliance) se usa TCP o RELP para garantizar entrega.
EJ-02 ¿Por qué la amplificación es tan efectiva? Fácil

Explicá el mecanismo de la amplificación UDP paso a paso, usando NTP monlist como ejemplo. ¿Por qué el atacante necesita muy poco ancho de banda propio para generar un ataque devastador?

MECANISMO DE AMPLIFICACIÓN NTP: Paso 1 — SPOOFING: El atacante falsifica el IP de origen del paquete → En su request pone src=IP_VÍCTIMA (en lugar de su IP real) → UDP no tiene handshake → el servidor no puede verificar el origen Paso 2 — REQUEST PEQUEÑO: El atacante envía "monlist" a un servidor NTP → Paquete de solo 46 bytes → Desde MILES de IPs spoofed (víctima como origen) Paso 3 — RESPUESTA MASIVA: El servidor NTP responde a la IP "origen" → monlist devuelve las últimas 600 conexiones del servidor NTP → Respuesta: hasta 25,564 bytes (556x más grande que el request) → El servidor NTP envía ESA respuesta A LA VÍCTIMA (no al atacante) Paso 4 — INUNDACIÓN: La víctima recibe 556x más tráfico del que el atacante generó, sin que el atacante necesite ese ancho de banda. POR QUÉ ES TAN EFECTIVO: → Multiplicación de recursos: 1 Mbps atacante = 556 Mbps hacia víctima → Anonimización: el servidor NTP es el que "ataca" directamente → difícil rastrear al atacante real → Infraestructura gratuita: millones de servidores NTP mal configurados en Internet sirven como amplificadores involuntarios → Sin defensa en el origen: el servidor NTP no sabe que está atacando SOLUCIÓN: BCP38 — los ISPs deben filtrar paquetes donde src_ip no corresponde al rango del cliente → elimina el spoofing en la fuente.

EJ-04 Cómo detectar si tu servidor NTP es un reflector Medio

Un compañero de trabajo dice "nuestro servidor NTP tiene picos de CPU y ancho de banda en horarios aleatorios". ¿Cómo verificás si está siendo usado como reflector en un ataque de amplificación? ¿Qué hacés si confirmás que sí?

VERIFICACIÓN — cómo confirmar que es reflector: 1. tcpdump en el servidor NTP: tcpdump -i eth0 -n 'udp port 123' -c 1000 → ¿Hay muchas IPs distintas enviando monlist requests? → ¿El src_ip de los requests parece spoofed (IPs de rangos remotos sin sentido)? 2. Verificar si monlist está habilitado: ntpdc -c monlist 127.0.0.1 → Si responde con lista de IPs = vulnerable → Si da error/timeout = ya está mitigado 3. Verificar logs NTP: grep "monlist" /var/log/ntp.log journalctl -u ntp | grep -i "monlist\|restrict" 4. Analizar tráfico de salida: iftop -i eth0 -f 'udp port 123' → ¿El tráfico SALIENTE en UDP/123 supera al entrante por mucho? → Si orig_bytes >> resp_bytes en Zeek = está amplificando CONFIRMADO — ¿qué hacés? Mitigación inmediata: → Actualizar NTP: apt upgrade ntp / yum update ntp (versiones ≥4.2.7p26 tienen monlist deshabilitado por defecto) → Si no podés actualizar de inmediato: ntpdc -c "monlist off" # deshabilita en caliente (no persiste) → En /etc/ntp.conf agregar: restrict default noquery nopeer nomodify notrap restrict 127.0.0.1 → Solo permite queries desde localhost → Agregar ACL en firewall: iptables -A INPUT -p udp --dport 123 -s RED_CORPORATIVA -j ACCEPT iptables -A INPUT -p udp --dport 123 -j DROP → Solo acepta sincronización desde tu red Notificación: reportar el abuso a abuse@isp del atacante (si podés rastrear desde los logs quién envió los monlist requests).

Si es propio: reiniciar con -U 0 (deshabilitar UDP): memcached -U 0 VERIFICACIÓN: echo -e "\x00\x00\x00\x00\x00\x01\x00\x00stats\r\n" | nc -u target 11211 Si responde → vulnerable
EJ-07 SSDP: identificar routers vulnerables en la red Medio

SSDP (UPnP) es un protocolo que permite a dispositivos domésticos autodescubrirse. ¿Cómo lo aprovecha un atacante para amplificación? ¿Qué dispositivos suelen ser vulnerables? ¿Cómo identificarías en el SOC si hay reflectores SSDP en la red corporativa? ¿Qué política aplicarías?

CÓMO FUNCIONA LA AMPLIFICACIÓN: 1. Atacante envía M-SEARCH request UDP/1900 con IP spoofed de la víctima 2. Dispositivo UPnP responde con lista de servicios (respuesta más grande que request) 3. Factor amplificación: ~30x (moderado, pero muy extendido) 4. Miles de routers domésticos y dispositivos IoT → botnet de reflectores DISPOSITIVOS VULNERABLES: - Routers hogareños (D-Link, Netgear, Linksys) con UPnP habilitado - Smart TVs, impresoras, consolas de videojuegos - Cámaras IP con UPnP activo DETECCIÓN EN RED CORPORATIVA: # Escanear UDP/1900 en la red interna: nmap -sU -p 1900 --script upnp-info 192.168.0.0/24 # En Zeek: tráfico SSDP saliente hacia internet zeek-cut proto id.orig_h id.resp_h id.resp_p < conn.log | awk '$2=="udp" && $5=="1900"' POLÍTICA: - Bloquear UDP/1900 hacia Internet en el firewall perimetral - Deshabilitar UPnP en todos los routers corporativos - Segmentar dispositivos IoT en VLAN separada sin acceso externo

Herramienta · Análisis

Zeek — análisis UDP

# Top destinos UDP (flood detection)
zeek-cut proto id.resp_h id.resp_p < conn.log \
  | awk '$1=="udp"' | sort | uniq -c | sort -rn | head

# Amplificación: resp_bytes >> orig_bytes desde UDP/123
zeek-cut proto id.orig_h id.orig_p orig_bytes resp_bytes < conn.log \
  | awk '$1=="udp" && $3=="123" && $4>1000'
